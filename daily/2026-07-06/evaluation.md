## 3. 核心流程是否闭合
闭合。用户主路径全部走通并经实测确认：
1. 首屏默认选中「结账静默回归」场景，顶部即显示红色 REGRESSED 横幅 + 统计（2 回归 / 0 改善 / 2 守住 / 0 容差内变化）+ 源码 diff 旁注（+3/-1，看起来无害）。
2. 点击验收矩阵中 PASS→FAIL 的行 → 右侧证据面板更新，显示基线 vs 当前的证据 delta（HTTP 200→500、成功页截图→错误页截图）+ 判定依据。
3. 左栏切换 4 个场景 → 横幅/统计/矩阵实时更新：场景 1 红色 REGRESSED、场景 2 黄色 IMPROVED、场景 3 绿色 HELD、场景 4 REGRESSED（含去噪）。
4. 展开「How it works」→ 4 步流程 + before/after 价值对比正常显示。
5. 「只看发生变化的项」开关可过滤矩阵。

## 4. 首屏是否讲清产品价值
清楚。首屏一眼能看到「源码 diff +3/-1、很小很绿，却判 REGRESSED」的强反差，直接点破「行级 diff 看不见、行为已回归」这一核心卖点；顶部一句话定位 + 统计卡 + 源码 diff 旁注共同把「解决什么」讲清。

## 5. 交互是否有断点

## 7. Demo 的最大问题
- 判定逻辑是刻意简化的确定性对比（状态翻转 + 数值阈值），真实产品最难的是**证据规范化与去噪**（截图抖动、时间戳/随机 id、环境噪声导致的假阳性回归）。Demo 用「容差内变化」标签与阈值示例演示了这一意识，但真实鲁棒性未被验证。
- 证据均为 mock，未接入任何真实证据来源（TryCase/CI），因此「接入面小」这一卖点在 Demo 中是叙述性的、非实测。
