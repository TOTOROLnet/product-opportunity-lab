## 1. 报告事实 vs 我的独立判断

| 维度 | 报告怎么说（参考资料） | 我作为创业者的独立判断 |
| --- | --- | --- |
| 核心趋势 | Agent 交付物从“结果文本”升级为“可验证证据（截图/录屏/日志/artifact）” | **认同，但补一刀**：报告停在“单次产出证据”。真正被低估的痛点不是“这次有没有证据”，而是“**下一次跑，之前被证明能用的东西是不是被悄悄改坏了**”。证据 bundle 目前是一次性、跑完即弃的孤儿，没人把它当成可版本化的基线。 |
| TryCase | 创建一次性环境、采集 proof，把“请你测试”变成 agent 工作流的一部分 | **认同它解决了“证据从无到有”**。但它只负责“生产一份证据”，不负责“把多份证据横向对比、判断是进步还是回归”。它是证据的**来源**，不是证据的**裁决/记忆层**。这里有明显空位。 |
| CircleChat | 独立 judge 一次性验收产物，高危动作人工审批 | **部分质疑**：judge 一次性验收，但 agent 工作是持续、反复重跑的。一次性 verdict 会随时间失效——今天判 PASS，明天另一个 agent 重跑把它跑挂了，judge 不会自动重判。而且报告自己也点出风险：“judge/审批只是浅层 UI”。要不浅层，judge 必须能对**证据的变化**做增量判断，而不是每次从头读一遍。 |
| 验收标准 | CircleChat 有“带验收标准的任务” | **认同其价值，但它只是起点**：验收标准如果不绑定到具体证据、且只在“完成时刻”判一次，就无法回答“它还稳不稳”。验收应该是**可持续复检的契约**，绑定证据、随每次重跑自动复判。 |
| 落地风险 | TryCase 风险在集成与成本；CircleChat 风险在替代既有工作流 | 认同。**这恰好说明创业切入点不该再造一个“环境/工作台”**（重、要接真实系统、迁移成本高），而应做一个**轻的、消费别人证据的裁决层**——不产环境、不抢工作台，只吃证据 bundle、吐回归 verdict。集成面小、价值密度高。 |

（我认同“证据层是主战场”，但**质疑“一次性证据/一次性 verdict”是终点**。我的独立判断是：证据的价值在于**跨时间对比**，缺失的是“验收回归”这一层。）

## 2. 候选机会（至少 3 个，均为创新切入点而非照抄）

### 候选 A：Datum — Agent 交付证据的验收基线与回归哨兵（Evidence Baseline & Acceptance-Regression Sentinel）
- 创新切入点：把 TryCase 式**一次性证据 bundle** 升级为**可版本化的基线**；对每次 agent 重跑，用“验收标准 ⇄ 证据”的绑定关系做**跨 run 的增量复判**，只暴露**发生变化 / 从通过变失败（回归）**的验收项与对应证据 delta。判定：HELD（守住）/ REGRESSED（回归）/ IMPROVED（改善）。
- 目标用户 / 场景：把真实任务托付给 coding agent / 业务 agent 的团队；用 TryCase / CI 之类工具已经在产证据，但缺一层“别悄悄改坏我之前证明能用的功能”的守门。
- 五维评分：痛点 5/5，机制新意 5/5，方向相关 5/5，一天可行性 5/5，启发价值 5/5 → 合计 **25/25**（自评偏乐观，正式取 **24/25**，机制新意计 4：视觉/快照回归测试已存在，我们的新意在“证据×验收标准×跨run裁决”的组合，而非从零发明 diff）。

### 候选 B：Contract — Agent 任务的“可验收契约”预检 / 验收标准可验证性 linter
- 创新切入点：CircleChat 靠“带验收标准的任务 + 独立 judge”，但 judge 再强也只与验收标准一样好。现实里验收标准多是“做得更好点/修一下 bug”这种 judge 无法客观核验的话术。本机会做**运行前**的验收标准 linter：把自然语言目标拆成**可机读、绑定证据、可测**的验收项（given/when/then + 每条需要哪种证据佐证），并对验收标准打“可验收性分数”（模糊度/可测性/覆盖度）。把验收质量**左移**。
- 目标用户 / 场景：用多 Agent / judge 流程的团队；planner 在派活前先把 DoD 变成机器可判的契约。
- 五维评分：痛点 4/5，机制新意 4/5，方向相关 5/5，一天可行性 5/5，启发价值 4/5 → 合计 **22/25**。

### 候选 C：Contract-Drift — MCP 工具契约漂移哨兵
- 创新切入点：报告里 DocsAlot（托管 MCP 检索）、MentionDrop MCP（区分读取工具 vs 需人类意图触发的动作工具）都在把能力做成 MCP 工具。MCP 服务端演进会让工具 schema 悄悄漂移（改参数名/删工具/改返回结构），静默打断依赖它的 agent。做一个**快照 MCP 工具契约 + 破坏性漂移告警 + 影响面（哪些 agent/flow 用到）**的哨兵。
- 目标用户 / 场景：维护/依赖 MCP 工具生态的开发者与平台。
- 五维评分：痛点 4/5，机制新意 4/5，方向相关 4/5，一天可行性 5/5，启发价值 4/5 → 合计 **21/25**（MCP 在今日报告里是次要信号，方向相关度略降）。

## 3. 最终选择

- 选中：**候选 A — Datum（Agent 交付证据的验收基线与回归哨兵）**，总分 **24/25**。
- 为什么选它（相对其他候选的优势）：
  1. **同时吃住今日两个最强信号**：TryCase 的“证据 bundle”和 CircleChat 的“验收标准 + judge”，且给出报告没想到的第三步——**跨时间**。B 只吃 CircleChat 一侧，C 吃的是次要信号。
  2. **是空位而非红海**：TryCase 产证据、CircleChat 一次性判，两者都没做“证据的记忆与回归裁决”。Datum 不与它们竞争，而是消费它们的产物、补上缺口。
  3. **演示最直观**：before/after 截图、HTTP trace 变化、验收项从 PASS→FAIL 的翻转，3 分钟能讲清“解决什么/怎么用/增量在哪”。
  4. **与本实验室既有产出层不同**（见第 5 节的“为何非照抄”，尤其相对 Attestor / Contour 的区别）。

## 4. 深入论证（针对选中机会）

1. 目标用户：已经开始把“真实、可重跑的任务”托付给 coding agent / 业务 agent 的工程团队与平台方；他们已用 TryCase / CI / e2e 在产证据，痛在“证据只证明了这一次”。
2. 核心痛点（真实、高频、可付费）：**静默回归**——agent 重跑或换版本后，某个此前被证明能用的关键路径悄悄坏了（结账 500、登录按钮变 404、关键接口延迟飙升、测试从 12 过变 10 过），而源码 diff 看起来“很小很绿”，人不会逐帧重看整份证据，于是回归漏网到线上。可重复发生、代价高、愿意付费买“别让我漏掉回归”。
3. 现有产品为什么没解决好：TryCase 只产单次证据、不做跨 run 裁决；CircleChat 的 judge 一次性验收、不随重跑自动复判（报告自曝“judge 可能只是浅层 UI”）；传统视觉/快照回归测试是“测试工程视角、按用例”，不是“**Agent 交付视角、按验收标准绑证据**”，也不理解“这次是 agent 的第 N 次重跑”。
4. 为什么适合做轻量 Demo：判定逻辑是纯前端可完成的确定性对比（证据字段对比 + 验收项状态翻转），全部可 mock；无需真实环境、后端、外部 API。用“交互式 run 对比控制台 + 多场景”即可让人 3 分钟看懂增量价值。

## 5. 不照抄声明（四项必填）

1. **创新切入点是什么**：把“一次性证据 bundle”变成“可版本化基线”，用“验收标准 ⇄ 证据”的绑定做**跨 run 的增量复判**，只暴露变化项与回归项（PASS→FAIL），给出 HELD / REGRESSED / IMPROVED 的持续 verdict 与“回归条数”。
2. **解决的具体问题 / 场景**：agent 重跑 / 换版本导致**曾被证明能用的关键路径静默坏掉**，而源码 diff 看起来很小很绿、人不会逐帧重看证据 → 回归漏到线上。
3. **相对被参考产品的增量价值（差异点）**：
   - 相对 **TryCase**：TryCase 是证据的**来源**（跑环境、产 bundle）；Datum 是证据的**记忆 + 裁决层**（吃 bundle、存基线、判回归）。不产环境、不接真实系统，集成面极小。
   - 相对 **CircleChat**：CircleChat 一次性验收 + 是整套团队工作台；Datum 只做一件事——让验收**随每次重跑自动复判**，并把 judge 的工作从“重读整份证据”降为“只审变化的 delta”。
4. **为什么这不是照抄 X（本质区别）**：
   - 非 TryCase/CircleChat 克隆（如上）。
   - 非本实验室 **Attestor** 克隆：Attestor 是**单次、运行后**核验 agent 自述**文字声明** vs 证据；Datum 忽略文字声明，做的是**证据 vs 证据的跨 run 行为对比**（时间维度），对象与生命周期位置都不同。
   - 非本实验室 **Contour** 克隆：Contour 解析**源码类结构化产物**（IaC/schema/feature tree）做**语义 diff**，问“这次改动是否破坏设计意图”；Datum 对比的是**运行期行为证据**（截图/HTTP/日志/测试结果），问“**行为是否回归**”。输入、问题、判据都不同。
   - 非通用视觉/快照回归测试克隆：那些以“测试用例”为中心、面向测试工程；Datum 以“**Agent 交付的验收标准 + 证据**”为中心，面向“把任务托付给 agent”的验收信任场景，并以 run 为版本轴。

## 6. 不确定性与风险

- 需求侧风险：团队可能觉得“已有 CI/e2e 回归就够了”。反驳点是 Datum 的对象是 **agent 交付验收**（按验收标准绑证据、按 run 版本化、面向“托付信任”），而非按测试用例；但需要清晰话术区分，否则易被当成“又一个回归测试”。
- 落地 / 技术风险：真实产品需要**证据规范化 + 稳健的 diff/去噪**（截图抖动、时间戳、随机 id 会造成假阳性回归）。Demo 里我用带阈值/容差的判定与“可解释的 delta”来演示这一点，但真实工程量不小。
- 我可能判断错的地方：也许头部团队会等 TryCase / CircleChat 自己长出“跨 run 回归”功能，把这层内化；那样独立产品的窗口会变窄。缓解：Datum 定位为**跨工具的中立裁决层**（吃任意来源的证据），这正是被单一平台内化时的差异化。
