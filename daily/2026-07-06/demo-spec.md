# Demo 设计说明 — 2026-07-06

## 1. Demo 名称
**Datum** — Agent 交付证据的验收基线与回归哨兵（Evidence Baseline & Acceptance-Regression Sentinel）

## 2. 一句话定位
把 agent 一次性的“证据 bundle”变成可版本化的**基线**，对每次重跑做“验收标准 ⇄ 证据”的**跨 run 增量复判**，只把 **PASS→FAIL 的回归**和变化的证据 delta 顶到你面前。

## 3. 目标用户
把真实、可重跑的任务托付给 coding agent / 业务 agent 的工程团队与平台方；他们已用 TryCase / CI / e2e 之类工具在产证据，痛在“证据只证明了这一次”。

## 4. 使用场景
一个 agent 任务（如“修复结账流程”）此前有一份被 judge 判过 PASS 的**基线 run**（含截图/HTTP trace/日志/测试结果）。今天 agent 重跑或换版本产出**当前 run**。Datum 加载两份 run，对每条验收标准复判，告诉你：守住了 / 出现回归（哪几条、坏在哪条证据）/ 有改善。人只需审阅“变化的部分”，而不是重看整份证据。

## 5. Demo 策略分型（重要）

- 本机会属于：**■ 可视化/交互类产品**（它本身就是一个人看的裁决控制台；同时对“证据/基线”这类抽象对象用**价值可视化 + before/after 对比**呈现）。
- 采用的演示方式：
  - 直接做**可点击的 run 对比控制台**：选场景 → 看验收矩阵（基线 vs 当前，逐条状态翻转）→ 点某条验收项看**证据 delta**（截图 before/after、HTTP trace 变化、日志新增错误、测试结果变化）。
  - 用 **before / after 对比** + **mock 证据可视化** 直观展示增量价值；用一个“源码 diff 看起来很小很绿”的旁注强调“行级看不见、行为已回归”的核心卖点。
- 演示的是**我们分析出的创新切入点**（跨 run 的验收回归裁决），不是 TryCase / CircleChat 的克隆，也不是通用回归测试工具。

## 6. 核心流程
（用户从进入到“看懂价值”的主路径，3 分钟内可完成）
1. 首屏给出一句话定位 + 一个默认场景已选中的**验收矩阵**：顶部是总 verdict（HELD / REGRESSED / IMPROVED）+ 回归条数 + 意图“行级 diff 很小”的对照旁注。
2. 用户在左侧**场景选择器**切换 3–5 个 mock 场景（静默回归 / 改善 / 守住 / 混合），矩阵与 verdict 实时更新。
3. 用户点某条验收标准，右侧**证据 delta 面板**展开该验收项绑定的证据在“基线 vs 当前”的具体差异（截图/HTTP/日志/测试），并解释判定依据（含阈值/容差，说明如何避免抖动假阳性）。
4. 底部“How it works / before-after”条：before（judge 每次重读整份证据、回归靠人肉发现）→ after（Datum 只顶变化 + 回归自动 trip）。

## 7. 页面结构（<= 3 个主要页面）
单页应用（SPA），由三个主区域组成，无路由跳转，降低复杂度：
- 区域 1：**场景选择器 + 全局 verdict 头**（左栏 + 顶部横幅）。
- 区域 2：**验收矩阵**（每条验收标准：基线状态 → 当前状态，回归高亮，可点击）。
- 区域 3：**证据 delta 面板**（选中验收项后展开：截图 before/after、HTTP trace diff、日志 diff、测试结果 diff、判定说明）。
- 附：可折叠的 **How it works / before-after** 说明区。

## 8. 关键交互
- 场景切换：点击左栏场景 → 矩阵、verdict、旁注全部更新（纯前端状态）。
- 验收项选择：点击矩阵中一行 → 右侧证据面板显示该项的证据 delta；回归项默认高亮并自动选中第一条回归。
- 证据类型标签切换：截图 / HTTP / 日志 / 测试，用 tab 或分组展示。
- “只看变化”开关：切换“显示全部验收项 / 只显示发生变化的项”，直观体现“把 judge 的工作从重读整份降为审 delta”。

## 9. 使用的 mock 数据
（说明数据来源与结构，全部为 mock，不接真实后端/外部 API）
- `scenarios.ts`：4 个场景，每个含：任务名、源码 diff 摘要（行数，用于“看起来很小很绿”的旁注）、基线 run 与当前 run 元信息、若干**验收标准**。
- 每条验收标准结构：`{ id, text, evidenceType, baselineStatus, currentStatus, evidence: { baseline, current, note } }`；`evidenceType ∈ screenshot | http | log | test`；截图用**内联 SVG/纯 CSS 画的伪截图**（无外部图片），HTTP/日志/测试用结构化文本对比。
- 判定逻辑 `logic/verdict.ts`：由每条 `baselineStatus→currentStatus` 汇总出全局 verdict 与回归条数；含一个简单的“阈值/容差”示例（延迟类证据用数值 + 容差判定）。
- 4 个场景覆盖范围：
  1. **静默回归（主打）**：结账流程，源码 diff 仅 +3/-1 行、看起来很绿，但 HTTP `POST /checkout` 由 200→500、截图从“下单成功页”变“错误页”、日志新增 stripe 异常 → **REGRESSED**，2 条回归。
  2. **改善**：此前 flaky 的测试从 fail→pass、延迟从 1200ms→180ms 回落到容差内 → **IMPROVED**，0 回归。
  3. **守住（绿色路径，防止危言耸听）**：一次纯重构，所有验收项 PASS→PASS、证据行为一致 → **HELD**，0 回归。
  4. **混合 + 阈值边界**：一条真回归（登录按钮截图 404）+ 一条“看似回归实为抖动”（延迟 210ms 仍在 200ms±30 容差内，判 HELD 并解释为何不算回归）→ **REGRESSED**，1 回归，演示去噪。

## 10. 明确不做什么
- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实测试环境。
- 不真的跑 agent、不真的采集证据（证据全部 mock）。
- 不做生产级后端；不超过“单页 + 三区域”的复杂度。

## 11. 成功体验标准
- 用户 3 分钟内能说出：**它解决什么**（agent 重跑后的静默回归）、**怎么用**（选场景 → 看验收矩阵 → 点开证据 delta）、**增量在哪**（把一次性证据升级为可版本化基线 + 跨 run 只审变化、回归自动 trip）。
- 首屏即能看到“源码 diff 很小很绿，但行为已回归”的反差，卖点一眼可见。
- 至少一个绿色“守住”场景，证明 Datum 不是危言耸听、能分辨真回归与抖动。
