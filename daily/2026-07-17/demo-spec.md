# Demo 设计说明 — 2026-07-17

## 1. Demo 名称
常青 Evergreen — 活文档鲜度守护 agent

## 2. 一句话定位
把"鲜度"变成活文档的一等属性：agent 将每条承重论断绑定到真实来源，来源一变就精确到句检出静默失真、生成最小补丁，人一键采纳，文档鲜度分实时回升。

## 3. 目标用户
维护"活文档"的运营 / 客服 / PM / 合规 / 技术写作者。Demo 以中小 SaaS/电商的 **客服退款 SOP** 负责人为主角。

## 4. 使用场景
退款 SOP 依赖多个真实来源（定价表、退款政策、支付网关 API、财务对账窗口、物流时效）。这些来源在过去数周悄悄变了，SOP 仍写着旧值，一线客服照着旧 SOP 处理 → 退错款 / 违反新政策。常青持续盯住来源，来源变化时定位到 SOP 里失真的那几句、给出补丁，让负责人一键修复。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / 基础设施类产品（一个"持续维护"的 agent + 文档鲜度层，无直接可视实体）
- 采用的演示方式：**纯前端"模拟体验 + 价值可视化"**
  - **活文档视图**：把 SOP 渲染成带**逐句鲜度高亮**的文档（绿=鲜 / 橙=有风险 / 红=已腐），配**文档鲜度分**仪表。
  - **来源时间线（模拟核心）**：把"来源随时间变化"做成一条可**逐步推进 / 一键播放**的时间线；推进即触发对应论断变腐——这是"模拟体验"，让用户亲手制造漂移、亲眼看到静默失真被点亮。
  - **补丁评审（before/after）**：agent 提议的最小补丁以 diff 呈现（旧句→新句 + "为什么：来源 X 从 a 变成 b"），逐条采纳；采纳后文档变鲜、鲜度分回升——直观的价值可视化。
- 演示的是**我们分析出的创新切入点**（写入之后的鲜度维护），不是 Tiptap（写入可靠性）/ Mantle（cap-table 入表）/ Notion 的克隆。

## 6. 核心流程
（用户从进入到"看懂价值"的主路径，3 分钟内可完成）
1. 进入**文档视图**：看到一份"看起来正常"的退款 SOP，但鲜度分只有 ~58%，若干句被标红/橙——第一秒就制造认知冲击："我以为它是对的。"
2. 点开任一红句：右侧显示它绑定的来源、上次核验时间、来源现在变成了什么——理解"为什么腐"。
3. 切到**来源时间线**：一键播放，看着过去 30 天的来源变更事件逐个发生，文档里对应的句子实时变腐（亲手制造漂移，理解静默性）。
4. 切到**补丁评审**：逐条采纳 agent 的最小补丁（或看到"需人判断"的例外），文档变鲜、鲜度分从 58% 升到 100%（承重项）——看懂增量价值。

## 7. 页面结构（<= 3 个主要页面，用顶部 Tab 切换）
- 页面 1：**文档 · Document**（活 SOP + 逐句鲜度高亮 + 鲜度分仪表 + 点击句子看来源侧栏）
- 页面 2：**来源 · Sources**（绑定的来源列表 + 各来源当前值 + 可推进/播放的变更时间线；推进即改变文档鲜度）
- 页面 3：**补丁 · Patches**（agent 提议的最小补丁 diff + 采纳/驳回 + before/after 鲜度分 + "需人判断"例外）

## 8. 关键交互
- 时间线"推进一步 / 全部播放 / 重置"：真实改变来源当前版本 → 引擎重算每条论断鲜/腐 → 文档与鲜度分同步变化。
- 句子点击：高亮 + 联动右侧来源卡片（论断↔来源双向绑定）。
- 补丁"采纳"：把论断的断言值更新为来源新值 → 该句转绿 → 鲜度分回升；"驳回"保留红句。
- "需人判断"例外：某来源以模糊方式变化（如政策口径变含糊），引擎**不自动改写**，标为需人决策——诚实呈现能力边界。
- 一键"全部采纳可自动修复项"：演示"3 分钟修好一份腐烂文档"的爽点。

## 9. 使用的 mock 数据
（全部 mock，不接真实后端 / 外部 API）
- `sources.ts`：5 个来源，每个含**有序版本列表**（值 + 时间戳），例如退款政策 v1=14 天 → v2=7 天。
- `claims.ts`：SOP 正文按段落/句子拆成论断，每条含：句子模板（引用某来源的值）、绑定来源 id、是否承重、当前断言值、所属小节。
- `engine.ts`（真实确定性逻辑，非造假）：给定时间线位置（各来源当前版本）→ 逐条判定论断鲜/腐（断言值 vs 来源现值）→ 用来源新值套模板生成补丁文本 → 计算鲜度分（承重且鲜 / 承重总数）。
- `timeline.ts`：过去 30 天的来源变更事件序列，用于"推进/播放"。
- 数据来源说明：SOP 文本、来源值、变更事件均为围绕"退款处理"场景**虚构**的示例，用于演示机制，不代表任何真实公司。

## 10. 明确不做什么
- 不做登录 / 用户系统
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实 LLM（"论断抽取/语义判定"在真实产品里由 LLM 完成，Demo 中以预编码的绑定关系 + 确定性值比较代替，并在自评里如实说明）
- 不做生产级后端
- 不做真实的多来源爬取 / webhook 监听（用可推进的 mock 时间线代替）

## 11. 成功体验标准
- 用户 3 分钟内能说出："它解决什么（活文档静默失真）、怎么用（绑来源→推时间线看变腐→一键采纳补丁）、增量在哪（把鲜度变成可测量、可守护、可一键修复的属性）。"
- 看到鲜度分从 58%（承重项）随采纳补丁升到 100%，且理解有一条"需人判断"的例外没有被强行自动修——感知到产品的诚实与能力边界。
