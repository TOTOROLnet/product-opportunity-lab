## 2. 候选机会（至少 3 个，均为创新切入点而非照抄）

### 候选 A：常青 Evergreen — 活文档鲜度守护 agent
- 创新切入点：不做"帮你写文档"，也不做"审查 agent 改得对不对"，而是把**"鲜度（freshness）"变成活文档的一等属性**：
  agent 把文档里每条**承重论断**（价格、时限、政策、API 版本、SLA 等）绑定到它依赖的**真实来源**；
  当来源发生变化，agent 精确定位到**具体那句话**、生成**最小改写补丁**、并维护一个**文档鲜度分**。
  人只需 review-before-apply 一键采纳（复用报告的交互原则），无法自动修的则升级为"需人判断"。
- 目标用户 / 场景：运营/客服/PM/合规/技术文档负责人——任何维护"活文档"的人：
  客服 SOP、退款政策、onboarding runbook、产品规格、内部 wiki、供应商条款页。
- 五维评分：痛点 5/5，机制新意 4/5，AI 核心度 5/5，一天可行性 4/5，启发价值 5/5 → 合计 **23/25**

### 候选 B：领航 Helm — 文档即双向控制面（steer-by-document）
- 创新切入点：把一份"计划文档"变成正在运行的 agent 的**活指令**——你在文档里改一行，
  agent 就实时重排它排队中的动作（不是事后审批，而是**前向改向**）。riding Campus"共处同一工作面"。
- 目标用户 / 场景：用 agent 跑长任务的运营/工程，想用"编辑文档"而非"再开一轮对话"来纠偏。
- 五维评分：痛点 4/5，机制新意 4/5，AI 核心度 4/5，一天可行性 3/5，启发价值 4/5 → 合计 **19/25**
- （诚实短评：要诚实演示"正在跑的 agent 随文档改向"，纯前端只能脚本化回放，可信度打折；
  且"改向/纠偏"离治理族偏近，新意受我自身重复拖累。）

### 候选 C（2C 对照）：常读 Freshread — 个人"长青指南"守护
- 创新切入点：把候选 A 的"鲜度维护"下沉到 2C——帮个人维护会过期的私人指南
  （旅行攻略、装机清单、考公/考证复习提纲、育儿计划），当外部事实变化时提醒并改写。
- 目标用户 / 场景：内容创作者、学生、生活规划者。
- 五维评分：痛点 3/5，机制新意 2/5，AI 核心度 3/5，一天可行性 3/5，启发价值 3/5 → 合计 **14/25**
- （诚实短评：2C 的"来源变化"多依赖真实外部 API（价格/库存/政策），纯前端只能造假数据，
  说服力弱；且个人笔记类拥挤、留存假设薄。列入以证明我**认真考虑过 2C**，但按分数不选。）

## 3. 最终选择

- 选中：**候选 A — 常青 Evergreen（活文档鲜度守护 agent）**，总分 **23/25**
- 为什么选它（相对其他候选的优势）：
  1. **痛点最真最普遍**：文档会"静默腐烂"是每个组织的隐性成本——SOP 写着旧退款时限、runbook 指向已废弃的 API、
     onboarding 教着过时的流程；错误昂贵且**没人会主动发现**，因为文档"看起来"没变。这是高频、可付费的真痛点。
  2. **机制族全新**：这是我 15 天里**从未做过**的机制族（知识鲜度 / 活文档维护），
     明确避开了我过度重复的"审查 agent 行为/治理控制面"族——审计族审的是**agent 做得对不对**，
     常青维护的是**世界变了、文档还没跟上**，主体、用户、产出都不同（见第 5 节）。
  3. **AI 核心度满分**：论断抽取、来源变化的语义判定、最小补丁生成，都必须靠模型，是 AI-native 能力。
  4. **可诚实地做纯前端 Demo**：用 mock 文档 + mock 来源 + 可"推进"的时间线，
     真实实现"来源版本 → 论断鲜/腐判定 → 补丁生成 → 鲜度分"这条确定性逻辑，价值以 before/after 可视化。
  5. **启发价值最高**：它把"文档即 agent 界面"的趋势从"agent 会写"推进到"agent 会**养**"，
     并把 Mantle 被低估的"回链原文"泛化成一个横向能力——这对 AI 产品从业者是有增量的判断。

## 4. 深入论证（针对选中机会）

1. 目标用户：维护"活文档"的运营/客服/PM/合规/技术写作者。以中小 SaaS/电商的**客服退款 SOP**为首个楔子场景——
   这类文档改动频繁、被一线员工/客服 agent 每天引用、写错直接造成资损或合规风险。
2. 核心痛点（真实、高频、可付费）：活文档与它依赖的真实来源**持续漂移**，
   但漂移是**静默的**——价格表调了、退款政策从 14 天改成 7 天、支付网关废弃了一个接口、物流时效变了，
   而 SOP 还写着旧值。没人会逐句核对，直到出错才发现。**"文档看起来没变"正是它最危险的地方。**
3. 现有产品为什么没解决好：
   - Tiptap AI Toolkit 解"让 agent 可靠地写进文档"——解决**写入这一刻**，不管写完后是否失真。
   - Notion/Google Docs 的 AI 是**按需生成/改写**，你不问它不动，且不知道每句话依赖哪个来源。
   - Mantle Clerk 有"数值回链原文"，但它是 cap-table 垂直、且用于**入表抽取+拦无效操作**，
     不做"来源变化后跨文档的持续维护"。
   - 传统做法（定期人工 review、加"最后更新时间"）无法定位**是哪一句、因为哪个来源变了、该怎么改**。
4. 为什么适合做轻量 Demo：核心价值可完全用**确定性逻辑 + 可视化**表达——
   论断↔来源绑定表、来源版本时间线、鲜/腐判定、补丁模板生成、鲜度分，全部可在纯前端 mock 实现；
   "推进时间线看文档变腐、一键采纳补丁看鲜度回升"是 3 分钟内看懂价值的强交互。

## 5. 不照抄声明（四项必填）

1. 创新切入点是什么：把**"鲜度"做成活文档的一等、被 agent 持续维护的属性**——
   论断绑定真实来源、来源变化时**精确到句**地检出静默失真、生成**最小补丁**、维护**文档鲜度分**，
   人 review-before-apply 一键采纳。
2. 解决的具体问题 / 场景：活文档（客服退款 SOP、runbook、政策页、规格）随来源漂移而**静默失真**，
   无人发现直到出错——常青让"该改的那一句、为什么、改成什么"变得可见、可一键修复。
3. 相对被参考产品的增量价值（差异点）：
   - vs **Tiptap AI Toolkit**：Tiptap 解"写入的可靠性"（一次性动作）；常青解"写入之后的鲜度维护"（长期被动触发）。
   - vs **Mantle Clerk**：Mantle 是 cap-table 垂直、"回链原文"用于入表抽取；
     常青把"论断↔来源绑定"泛化为横向能力，且专用于**来源变化后的检出与自愈**。
   - vs **Notion/Docs AI**：它们按需生成，不追踪每句话的来源，也不主动告诉你哪句已腐。
   - 增量 = **把"文档鲜度"从没人负责的隐性成本，变成一个被测量、被守护、可一键修复的产品状态。**
4. 为什么这不是照抄（本质区别）：
   - 不是照抄 Tiptap/Mantle：对象不同（不是"帮你写/入表"，而是"帮你养、抵御失真"）。
   - 也不是我自己审查/治理族的换皮：**审计族审的是 agent 的行为/主张是否可信；常青审的是"世界变了、文档还没跟上"。**
     主体不同（外部来源 vs agent 行为）、用户不同（文档负责人 vs 监督者）、产出不同（自愈后的文档 + 鲜度分 vs 裁决/回滚）。

## 6. 不确定性与风险

- 需求侧风险：很多组织对"文档腐烂"痛而不自知，属于**教育型需求**，付费意愿需要用"避免了一次资损"来证明；
  首场景（退款 SOP）选得对不对决定楔子锋利度。
- 落地 / 技术风险：真实产品里"把哪句话绑到哪个来源""来源语义变化到什么程度算腐"高度依赖 LLM 抽取质量与去噪，
  会有误报/漏报；"最小补丁"在政策类文本上可能需要人判断而非自动改写（Demo 中我保留了一条"需人判断"的论断如实呈现）。
- 我可能判断错的地方：(a) Notion/Tiptap 等生态方可能顺手把"来源绑定+过期提醒"做成内建功能，挤压第三方；
  (b) "鲜度分"可能沦为虚荣指标，若检出不准会反噬信任；
  (c) 与我自己的漂移检测类开放点（Driftguard/RealityDiff）在"检出"机制上有**部分同韵**——
  我用"面向文档负责人的自愈产出 + 鲜度分"而非"面向工程的 diff/裁决"来做区隔，但需警惕滑回检测面板。
