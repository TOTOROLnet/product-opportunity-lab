## 2. 候选机会（至少 3 个，均为创新切入点而非照抄）

### 候选 A：随读（Catchup）— "我不在时，agent 把我的文档改了啥"的意图级简报
- 创新切入点：不做又一个 track-changes/审阅框，而是把 agent 对**共同编辑的活文档**的一堆改动，**按意图分组、按风险排序、用叙事讲清"改了啥/我该不该在意"**，让你在意图层（而非逐字）accept/undo，并显式暴露 agent"悄悄替你做的假设"。
- 目标用户 / 场景：越来越多用 Tiptap/Campus 这类"agent 直接写入文档"的人，离开一会儿回来面对一片改动。
- 五维评分：痛点 4/5，机制新意 3/5，AI核心度 3/5，一天可行性 5/5，启发价值 3/5 → 合计 **18/25**
- 备注：真实、可做，但**又是"审阅/控制"家族**——正是我近两周反复做的类型，边际启发低。

### 候选 B：师承（Shicheng）— 让领域专家"带教"出可复用、可检验的 Agent Skill
- 创新切入点：把"造一个 Skill"从**工程师写 prompt/工具**，反转为**领域专家用批改（红笔）来教**：agent 先对真实 case 给出判断+理由，专家只需"认同 / 改判 / 圈出错误理由并补一条规则"；系统把这些批改**蒸馏成一张人类可读、可检验、可版本化的技能卡（Skill）**，每条规则都带来源 case 与权重，可开关、可编辑、可导出上线。核心反转：**Skill = 被捕获的判断力，用批改而非写提示词来生产。**
- 目标用户 / 场景：手里有值钱隐性判断、但不会写 prompt/看不懂 markdown Skill 的领域专家——资深招聘官、客服主管、风控/合规、投资初筛人、医疗初诊分诊等；他们想把自己的判断"复制"进 agent，却卡在"不会造 Skill"。
- 五维评分：痛点 4/5，机制新意 5/5，AI核心度 4/5，一天可行性 5/5，启发价值 5/5 → 合计 **23/25**

### 候选 C：捏形（V2-adjacent）— 生成式 3D 角色的"可玩性优先"创作
- 创新切入点：不比拼建模质量，而是把 3D 角色创作重构为"先能玩/能动起来"的体验（文本改性格→即时看动作/表情反应）。
- 目标用户 / 场景：想快速产出"有生命感"角色的独立开发者/创作者。
- 五维评分：痛点 3/5，机制新意 3/5，AI核心度 3/5，一天可行性 2/5，启发价值 3/5 → 合计 **14/25**
- 备注：3D 品质是其全部价值，纯前端 mock **无法诚实演示**，价值可视化会造假；赛道拥挤。淘汰。

### （被显式放弃的方向）候选 X：agent 治理台（YAGNI 式）
- 虽是报告首推、且分数可能到 19–20，但与我 07-12 明约、07-14 值当、07-10 Pane 同族。**作为创业者主动放弃**：避免第 N 次做治理/审计台，保持组合创新度与边际启发。这本身是一次独立判断。

## 3. 最终选择

- 选中：**候选 B — 师承（Shicheng）**，总分 **23/25**
- 为什么选它（相对其他候选的优势）：
  1. **踩中一个我从没碰过、报告也没点破的信号**：报告把 Crustdata 讲成"护城河样板"，我看到的是"专家判断→Skill 的生产鸿沟"。这是"Skill 即产品"趋势的**供给侧**，几乎无人正面做。
  2. **是产品形态的反转，不是改良**：A 是给"审阅"换皮（增量），C 是拼 3D 质量（红海）；B 把"造 Skill"从"写提示词"反转成"批改带教"，是新的产品范式。
  3. **跳出我的偏科**：A 和被放弃的 X 都是"控制/审阅/治理"家族——我近 12 天几乎全是这类；B 是"判断力捕获"这一全新家族。
  4. **纯前端能诚实演示且价值可算**：我会实现一个**真实的迷你规则引擎**——每条被采纳的规则真的会改变 holdout 集上的判定，"带教前 vs 带教后 与专家一致率"是**实算出来的**而非编的数字。这让"价值可视化"诚实、可信，一天可行性最高。

## 4. 深入论证（针对选中机会）

1. 目标用户：**握有高价值隐性判断、但非工程师**的领域专家（资深招聘官/客服主管/风控/投资初筛/分诊等），以及想让 agent"按我的标准干活"的中小团队负责人。
2. 核心痛点（真实、高频、可付费）：企业想让 agent 替自己做判断，但 agent 默认的判断**不像我**——它会放过我一眼就否掉的候选、纠结我根本不在意的点。要修正它，今天只有两条路：**自己学写 prompt/Skill（专家不会、也没耐心）**，或**雇工程师翻译我的脑子（贵、失真、迭代慢）**。判断力是专有且值钱的资产，可"判断力→产品"的生产工具却缺位。这个痛点高频（每次想让 agent 上岗都撞上）、强付费意愿（判断力直接对应营收/风险）。
3. 现有产品为什么没解决好：
   - Crustdata 这类是**成品垂直 agent**——把判断力**替你**编码进 Skill，你无法用自己的判断去造/改，且只覆盖招聘一个域。
   - Prompt/Skill 编辑器、eval 工具（如 promptfoo/评测台）面向**工程师**：给你打分、给你 diff，但不帮**非工程师专家**把隐性判断变成可读规则；标注工具产出的是**训练数据**，不是可上线、可检验、可解释的 **Skill 产品件**。
   - 没人把"**批改 → 蒸馏成人类可读 Skill → 实测一致率提升**"这条闭环做成给专家用的产品。
4. 为什么适合做轻量 Demo：核心价值靠**交互闭环 + before/after**说清，不依赖真实 LLM。用 mock 语料（预置 case + 每条修正对应的规则）+ 一个**真实但极小的规则引擎**（规则命中 case 特征→改变判定），就能让人 3 分钟内体验"看 agent 判断→红笔批改→技能卡生长→holdout 一致率从低到高"，且这个提升是**真算出来的**。

## 5. 不照抄声明（四项必填）

1. **创新切入点是什么**：把"制造一个 Agent Skill"从"工程师写 prompt/工具"**反转为"领域专家用批改来带教"**——专家只做他最擅长的事（对具体 case 说"这样判/那样判、因为…"），系统把这些批改**蒸馏成人类可读、带来源、可版本化、可上线的技能卡**，并用 holdout 集上的"与专家一致率"实测其价值。
2. **解决的具体问题 / 场景**：非工程师专家想把自己的判断力注入 agent，却卡在"不会造/改 Skill"；今天只能自学 prompt 或雇工程师翻译（贵、失真、慢）。师承让专家**不写一行 prompt**，靠批改就能产出并持续打磨一个"像我一样判断"的 Skill。
3. **相对被参考产品的增量价值（差异点）**：
   - vs **Crustdata Recruiter**：它是**招聘域的成品 agent**（判断力被工程师替你编码）；师承是**领域无关的 Skill 生产工具**（判断力由**你自己**通过批改注入），招聘只是我 Demo 里的一个示例域。
   - vs **prompt/eval/标注工具**：它们面向工程师、产出分数/diff/训练数据；师承面向**专家**、产出**可读可上线的 Skill 产品件**，并把"一致率提升"作为核心指标闭环。
   - 新增：**带来源的可解释规则**（每条规则回链到教它的那个 case，呼应 Mantle Clerk"数值回链原文"的可信思路，但用在**判断规则**上）、**规则开关的实时一致率对比**、**批改→蒸馏**的授权式生产流程。
4. **为什么这不是照抄 Crustdata / 现有工具**：Crustdata 的产品终点是"一个能筛简历的招聘 agent"；师承的产品终点是"**专家的判断力被复制成一张可检验、可上线、可交易的 Skill**"。二者出发点（都涉及判断力+Skill）相似，但**用户（专家 vs 工程师）、核心交互（批改带教 vs 写 prompt）、产出物（可读 Skill 产品件 vs 垂直 agent/训练数据）、成功指标（与专家一致率提升）完全不同**。用户用师承不是为了"筛一次简历"，而是为了"把我的判断力变成一个能替我干活、还能拿去卖的 Skill"。

## 6. 不确定性与风险

- 需求侧风险：也许多数专家**不愿意/没耐心批改足够多 case**（冷启动成本），或觉得"我自己判还快些"；需要验证批改 N 条后一致率跃升是否足够快到让专家觉得"值"。
- 落地 / 技术风险：真实产品里"从批改**可靠蒸馏出正确、可泛化、不互相冲突的规则**"依赖 LLM 质量，且规则冲突/过拟合是硬问题；Demo 用预置规则+迷你规则引擎回避，真实上线需解决蒸馏质量与规则治理。
- 我可能判断错的地方：也许"Skill 即产品"更多仍会由工程师+数据护城河驱动（Crustdata 路径），专家自助造 Skill 是伪需求；若如此，师承的价值应下沉为"给工程师+专家协作用的判断捕获层"，而非纯专家自助。需要用真实专家的批改留存与 Skill 复用率证伪。

- 最高分：23/25（候选 B 师承 Shicheng）
- 是否达到 16/25 门槛：**是**
- 结论：**进入 Demo 开发**（Loop 2–5）。
