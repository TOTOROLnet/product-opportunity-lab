# Demo 设计说明 — 2026-07-16

## 1. Demo 名称

**师承（Shicheng）** — 让领域专家用"批改"带教出可复用、可检验的 Agent Skill。

## 2. 一句话定位

不写一行 prompt：领域专家只需对 agent 的判断"认同 / 改判 / 圈错补规则"，系统把批改**蒸馏成一张人类可读、带来源、可上线的技能卡**，并用 holdout 集"与专家一致率"**实测**这次带教值不值。

## 3. 目标用户

握有高价值隐性判断、但不会写 prompt / 看不懂 markdown Skill 的领域专家（Demo 用"资深后端招聘官"为示例域），以及想让 agent "按我的标准干活"的团队负责人。

## 4. 使用场景

专家想让 agent 替自己做初筛/判断，但 agent 默认"判得不像我"。专家打开师承，看 agent 对一批真实 case 的判断与理由，用红笔批改；随着批改积累，技能卡长出一条条带来源的规则，agent 在新 case（holdout）上"与我一致率"肉眼可见地上升。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / 工具 / 基础设施类产品（"判断力捕获 → Skill 生产"的元工具）——**同时具备可交互性**。
- 采用的演示方式：**模拟体验 + 价值可视化 + 真实迷你规则引擎**。
  - 交互：真实的"看判断 → 批改 → 技能卡生长"闭环（不是录像回放，是可点击操作）。
  - 价值可视化：**before / after 一致率对比**——关键点是这个数字**由一个真实的极小规则引擎实算**：每条被采纳的规则真的会改变 holdout 集上每个 case 的判定，采纳越多、与专家一致率越高。**不造假数字。**
- 演示的是**我们分析出的创新切入点**（专家批改带教 → 可读 Skill），不是 Crustdata（成品招聘 agent）或 prompt/eval 工具的克隆。

## 6. 核心流程（3 分钟内看懂"解决什么/怎么用/增量在哪"）

1. **首屏讲价值**：一句话 + before/after 一致率仪表（初始一致率低，例如 ~50%），点明"agent 判得不像你"。
2. **带教工作台（Coach）**：逐个看 case（候选人卡片：结构化字段 + 简述）→ 看 agent 的判定（通过/待定/拒）+ 一句理由 → 专家操作：
   - 「认同」：确认这条判断；
   - 「改判」：改变结论；
   - 「圈错理由 + 补一条规则」：从系统给的候选规则里采纳（或编辑）一条，规则**动画飞入技能卡**。
3. **技能卡（Skill）**：实时生长的 Skill——一条条人类可读规则，每条带**来源 case**、权重/效果、开关、可编辑。这就是"可上线/可交易的产品件"，提供"导出 Skill（JSON/Markdown）"按钮（前端下载，非真实后端）。
4. **验收（Evaluate）**：对 holdout 集"运行当前技能卡"→ 展示**带教前 vs 带教后**与专家一致率、以及"哪条规则修好了哪些 case"的归因；支持逐条开关规则实时看一致率变化（证明每条规则真的在起作用）。
5. 收尾：一句"你的判断力已被复制成一张 Skill"，回扣定位。

## 7. 页面结构（<= 3 个主要页面）

单页应用，三个主视图 Tab（均为主要页面）：
- **Coach（带教工作台）**：case 流 + agent 判断 + 批改操作 + 右侧实时技能卡预览。
- **Skill（技能卡）**：完整技能卡详情，规则来源/权重/开关/编辑/导出。
- **Evaluate（验收对比）**：holdout 一致率 before/after 仪表、规则归因、逐条规则开关联动。

## 8. 关键交互

- 批改三选一（认同 / 改判 / 圈错补规则），采纳规则时有"飞入技能卡 + 一致率数字跳动"的即时正反馈。
- 技能卡规则可开关/编辑，Evaluate 页一致率**随开关实时重算**（凸显"规则真的在起作用"）。
- 顶部常驻一个"当前与专家一致率"进度环，贯穿全程强化 before→after 叙事。
- 一键"重置带教"，方便重复体验。

## 9. 使用的 mock 数据

全部为 mock，不接真实后端/LLM/外部 API：
- **域配置**：招聘岗位（高级后端工程师）与几条"专家在意/不在意"的隐性偏好（用于生成 ground truth）。
- **训练集 case（~10）**：每个含结构化特征（年限、是否有生产级后端、开源贡献、是否夸大自述、领域匹配等）+ 简述；每个带 agent 初始判定+理由、专家 ground truth、以及"该 case 应教出的规则"候选。
- **holdout 集 case（~8）**：含特征与专家 ground truth，用于实算一致率。
- **规则引擎**：每条规则是 `(特征谓词) → 判定调整`；对每个 case：基线 agent 判定 + 已激活规则 → 最终判定，与 ground truth 比对得一致率。数据结构在 `src/data/` 与 `src/logic/`，纯确定性、可复现。

## 10. 明确不做什么

- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实 LLM / 真实招聘数据 / ATS。
- 不做真实"从批改蒸馏规则"的 NLP（Demo 用预置候选规则模拟蒸馏结果）。
- 不做生产级后端；"导出 Skill"仅为前端文件下载。
- 不超过 3 个主要页面；全部数据 mock。

## 11. 成功体验标准

- 用户 3 分钟内能说出："它解决什么（专家不写 prompt 就能把判断力变成 Skill）、怎么用（看判断→批改→技能卡生长）、增量在哪（一致率实测上升、规则可读带来源、领域无关的生产工具而非成品垂直 agent）"。
- **诚实性**：一致率是真算的；关掉规则一致率会掉回去——让用户相信这不是编的数字。
- 首屏不空白，核心闭环可完整走通，交互无断点。
