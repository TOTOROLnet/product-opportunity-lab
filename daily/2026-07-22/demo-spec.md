# Demo 设计说明 — 2026-07-22

## 1. Demo 名称

付前一秒 CommitGuard — 你的 AI 消费 agent 与钱包之间的「成交守门人」

## 2. 一句话定位

当你的 AI agent 即将点下「确认支付」的前一秒，CommitGuard 把它要成交的那一单摊开成人话核验单——意图对齐、真实总价、陷阱雷达、可逆性评分——让你看清代价再决定放行 / 让它改 / 拒绝。

## 3. 目标用户

把「预订、订阅、比价下单」等要真花钱的任务交给 AI agent、但对 agent 乱花钱有本能恐惧的普通消费者（差旅党、订阅重度用户、跨境网购党）。

## 4. 使用场景

你给 agent 下了一个带约束的任务（例：「订下周五去上海的酒店，2 晚，≤¥800/晚，必须能免费取消」）。agent 跑完带回一批「即将成交」的订单，CommitGuard 逐单在**成交前**拦下核验，你 3 分钟内完成放行/改单/拒绝，并看到本次会话「省了多少钱、拦了几个陷阱」。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 可视化/交互类产品（成交前核验是一种可点击的交互体验）　□ 抽象/CLI/基础设施类产品
- 采用的演示方式：**直接做可点击交互 Demo + before/after 价值可视化**。
  - 交互核心：对每一笔 agent 提交的「待成交订单」逐单核验并做出决策（批准 / 让 agent 改 / 拒绝）。
  - 价值可视化：一个「无守门人 vs 有守门人」的战报页，量化省下的真实金额、拦下的陷阱数、避免的不可逆承诺。
- 演示的是**我们分析出的创新切入点**（消费者侧、成交前、站在人这边的授权闸门），**不是** CartAI（卖家侧清算层）的克隆。

## 6. 核心流程

（用户从进入到「看懂价值」的主路径，3 分钟内可完成）

1. 首屏读到：我把「订上海酒店、≤¥800/晚、必须免费取消」交给了 agent；agent 带回 4 笔待成交订单，CommitGuard 已在成交前全部拦下待你核验。
2. 点开第 1 笔（酒店）→ 看到核验单：意图对齐里「免费取消 ✗（不可取消特惠）」被标红；真实总价 ¥851 超预算且隐藏了 ¥152；陷阱雷达列出「预勾选 ¥88 旅游保险」。→ 点「让 agent 改」，附带修改指令。
3. 点第 2 笔（AI 工具订阅）→ 「首月 ¥1」背后是自动续订 ¥199/月、取消需邮件+3 工作日，可逆性 22/100。→ 点「拒绝」。
4. 点第 3 笔（特价机票）→ drip pricing 把 ¥450 抬到 ¥712、不可退改、票代可信度低。→ 点「拒绝」。
5. 点第 4 笔（另一家可免费取消的酒店）→ 意图全部 ✓、真实总价透明无隐藏、可逆性 92/100、无陷阱，CommitGuard 建议「可放心批准」。→ 点「批准成交」。
6. 切到「价值对比」页 → 看到战报：若全部放行会踩中 3 类陷阱、多付 ¥X、留下 2 笔几乎不可逆承诺；用 CommitGuard 后省下 ¥X、拦下 N 个陷阱、只放行 1 笔干净的单。

## 7. 页面结构（<= 3 个主要页面）

- 页面 1：**守门台（Guard）**——顶部任务卡（委托给 agent 的任务 + 预算约束）+ 实时结余条；左侧为 agent 提交的「待成交订单」列表（含风险徽标），右侧为选中订单的**成交核验单**与决策按钮。
- 页面 2：**价值对比（Impact）**——「无守门人 vs 有守门人」before/after 对照 + 本次会话战报 + 拦截账本（逐条记录拦下了什么、为什么）。
- 页面 3：**它是什么（About）**——一页讲清创新切入点、与 CartAI 的本质区别、以及「这是纯前端模拟，不接真实支付/API」的诚实声明。

## 8. 关键交互

- 订单卡片点击 → 右侧载入对应核验单（意图 diff / 真实总价拆解条 / 陷阱雷达 / 可逆性仪表）。
- 三个决策按钮（批准成交 / 让 agent 改 / 拒绝）→ 订单状态更新，顶部「已省金额 / 已拦陷阱 / 已放行」实时累加。
- 「让 agent 改」→ 展示一条 CommitGuard 自动生成的、发回给 agent 的修改指令（模拟人机协作闭环）。
- 「价值对比」页的 before/after 数值由已做决策实时计算，凸显守门带来的增量。

## 9. 使用的 mock 数据

全部为 mock，硬编码在 `src/data/scenario.ts`，不接真实后端/外部 API/支付：

- 1 个委托任务（含预算与「必须免费取消」约束）。
- 4 笔待成交订单，每笔含：商户、品类、agent 填好的购物车、首屏标价、真实费用拆解项、意图对齐项（✓/✗）、陷阱清单（类型+严重度）、可逆性分项（退款窗口/难度/取消路径）、商户可信信号。
- 每笔订单预置一条「让 agent 改」的建议指令文案。
- 「核验引擎」（`src/logic/engine.ts`）为**确定性纯函数**：由拆解项算真实总价与隐藏金额、由陷阱清单算风险等级、由可逆性分项算 0–100 评分、汇总 before/after 战报。框架上写明「真实产品此处为 AI 解析结果」，Demo 用确定性 mock 代理。

## 10. 明确不做什么

- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实浏览器自动化。
- 不做真实的 dark pattern 识别模型（用 mock 标注代理，并在 About 页诚实说明）。
- 不做生产级后端；不超过 3 个主要页面；全部数据 mock。

## 11. 成功体验标准

- 用户 3 分钟内能说出：它解决什么问题（让人在 agent 成交前看清代价并可否决）、怎么用（逐单核验→放行/改/拒）、增量在哪（站在消费者一侧、成交前、跨商户，与 CartAI 相反）。
- before/after 战报能用具体金额与陷阱数，直观让人「想给自己的购物 agent 装一个」。
- 干净的单能被快速「放心批准」，体现它不是一味拦截、而是分级放行。
