# Demo 设计说明 — 2026-07-19

## 1. Demo 名称
并笔 CoBaton — 人机共写的「执笔权」协调引擎

## 2. 一句话定位
垫在任意编辑器之下的冲突消解层：人和 AI agent 在同一份活文档上并行编辑时，保证你永不被打断、你的编辑永不被静默丢弃、agent 的活儿永不白费。

## 3. 目标用户
用 Cursor / Claude Code / coding & doc agent 与 AI **同时改同一份文件**的开发者与知识工作者；以及要做"人机共编面"产品、需要一层协调底座的团队。

## 4. 使用场景
你正在改一份《退款服务 SOP》文档；同时给 agent 派了任务"把文档更新到与新版 API v2 一致"。你在 §2 打字，agent 想重写 §2、§4、§5。谁让路？agent 会不会覆盖你正在写的那句？会不会白重写你几秒后要删的 §5？CoBaton 负责仲裁这一切。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / CLI / API / 基础设施类产品（一个协议/引擎，不是可见的编辑器）
- 采用的演示方式：纯前端 **"模拟共写会话回放 + 玻璃盒所有权可视化 + before/after 对比"**：
  - **模拟共写回放**：脚本化一条真实感的共写事件序列（人移动焦点/打字 + agent 领任务/声明区域/编辑），可播放/单步。
  - **玻璃盒所有权地图**：把抽象的"区域所有权 + claim 队列 + 接力棒"可视化，用户看得见协调逻辑而非黑盒。
  - **before/after 对比**：同一条事件序列分别跑"朴素模式（last-write-wins / agent 自由编辑）"与"并笔模式（CoBaton）"，量化差异。
- 演示的是**我们分析出的创新切入点**（执笔权协调协议），不是被参考产品 OpenMarkdown（编辑器）的克隆。

## 6. 核心流程（用户 3 分钟内看懂价值）
1. 进入 **① 共写台**：看到一份分小节的《退款 SOP》文档，顶部有模式开关【朴素 ↔ 并笔】与【▶ 播放 / 单步 / 重置】。默认并笔模式。
2. 点【▶ 播放】：时间轴逐步推进，文档实时变化；每一步右侧浮出一句"刚发生了什么"（例：`agent 想改 §定价，但你正在 §定价 打字 → 已延后到你离开后再落改`）。人的活跃区高亮、agent 落改的小节高亮。
3. 切到 **朴素模式** 重放同一序列：直观看到你在 §2 的在写内容被 agent 覆盖（标红=丢失）、光标被迫跳走（被打断）、agent 白重写了你随后删掉的 §5。
4. 进 **② 执笔权地图（玻璃盒）**：看每个小节的所有者状态（你 / agent / 空闲）、pending claim、接力棒队列、逐事件日志；点小节看它的完整历史与"为何延后/为何再校验"。
5. 进 **③ 对比**：同序列两模式并排，量化 4 个指标：丢失的你的编辑数 / 被打断次数 / agent 白做的编辑数 / 最终文档一致性。并笔模式全 0，朴素模式非 0，价值一目了然。

## 7. 页面结构（<= 3 个主要页面 / Tab）
- 页面 1：**① 共写台（Co-writing stage）**——主体验：活文档 + 模式开关 + 时间轴回放 + 实时"发生了什么"解说。
- 页面 2：**② 执笔权地图（Ownership map / glass box）**——当前时间点的区域所有权状态、claim 队列、接力棒、逐事件日志。
- 页面 3：**③ 对比（Before / After）**——同序列两模式的可测指标并排对比。

## 8. 关键交互
- 模式开关：朴素 ↔ 并笔，同一事件序列即时重放，凸显差异。
- 时间轴：▶ 播放 / ⏭ 单步 / ↺ 重置；拖动/点击跳到任意事件。
- 活文档：人活跃区（蓝）、agent 落改区（绿）、丢失/被覆盖内容（红划除）、延后 claim（黄虚线）实时着色。
- 所有权地图：点小节 → 右侧详情（状态 + 事件历史 + 延后/再校验原因）。
- 对比页：两列指标卡 + 一句结论。

## 9. 使用的 mock 数据
（全部为 mock，不接真实后端/外部 API / 不接真实 agent）
- `data/doc.ts`：一份《退款服务 SOP》初始文档，5–6 个小节（标题 + 正文），每节带 id。
- `data/timeline.ts`：一条脚本化事件序列（12–16 个事件），事件类型含：
  `human_focus`（人聚焦某节）/ `human_edit`（人改某节文本）/ `human_blur`（离开）/
  `agent_task`（领任务）/ `agent_claim`（声明想改的节）/ `agent_edit`（对某节的目标改写）/ `human_delete`（人删掉某节）。
- 引擎对同一条序列在两种模式下归约，产出每一步的文档快照、所有权状态、指标——**数字由引擎算出，非硬编码**。

## 10. 明确不做什么
- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实 LLM / 真实 agent。
- 不做真正的富文本编辑器、不做 CRDT/OT 生产实现、不做本地文件存储（诚实标注：这是协议的**可视化模拟**，非生产实现）。
- 不超过 3 个主要页面；全部数据 mock。

## 11. 成功体验标准
- 用户 3 分钟内能说出：解决什么（人机共写撞车/丢编辑/被打断/白做）、怎么用（同序列切两模式看差异 + 玻璃盒看协调逻辑）、增量在哪（把"回避冲突"升级成"消解冲突 + 执笔权移交 + 再校验"，价值量化为 0 丢失 / 0 打断 / 0 白做）。
- 对比页的两模式指标差异真实来自同一引擎、同一序列，可信不造假。
