# 榫卯 Tenon —— 多 agent 共享工件的「接口契约 + 漂移哨兵」

给 **Blaxel Agent Drive** 这类多 agent 共享盘架一层「**类型系统 + 集成测试**」：
在生产者 agent 改坏工件的**形状 / 单位 / 语义**、下游 agent **静默误读**之前，
于**写入时**按契约拦截、算清**下游影响**、给出**最小迁移补丁**。

> 本项目为 product-opportunity-lab 2026-07-24 的每日机会 Demo。灵感来自当日报告
> A 类趋势 #2「Agent 运行时底座从『算力/隔离』扩到『共享状态/文件系统』」（代表作 **Blaxel Agent Drive**）——
> 但**不照抄**：Blaxel 解决的是「能并发读写的管道」，榫卯解决的是「管道里工件彼此**契合**的保证」。
> 二者关注点正交、可叠加。榫卯也**刻意不是**又一个 valv/Kastra/HOL Guard 式的**动作授权治理面板**——
> 它校验的是 agent 之间交换的**数据结构契约**，不做审批/权限/审计。

## 它解决什么问题

多 agent 通过共享盘交换中间工件（`findings.json`、`metrics.json`、`report.md`…）。
共享盘让「交接」变简单，却让「**误读**」变隐蔽：生产者 agent 一旦悄悄把某工件的字段改名、
把单位从 `0–100` 改成 `0–1`、删掉一个字段、或让枚举冒出新值，下游消费者 agent **不会报错**，
而是**继续跑、给出貌似合理却错误的结果**。错误在流水线里传播多步后，才以「报告数字离谱」暴露——
这是多 agent 系统里最贵、最难查的一类失败。榫卯把这类**静默运行时腐坏**，变成**写入时可捕获、
可解释、可迁移**的事件。

## 核心机制（确定性引擎，非套壳）

- **接口契约**：每个共享工件登记一份 schema 契约（字段 / 类型 / 单位 / 取值域 / 谁依赖）。
- **漂移哨兵**：`src/logic/engine.ts` 对「研究员发起的改动组合」判定是否破坏契约，
  并模拟下游分析师 / 报告员在「榫卯 OFF / ON」两种世界下的真实结局（含具体数字：
  `score=87` 在单位漂移下被当成 `0.87` → 综合置信塌成 `1/100` → 结论从「建议推进」反转为「不建议推进」）。
- **下游影响 + 最小迁移**：ON 世界给出违反了契约的哪一条、影响哪些下游 agent，
  以及最小迁移补丁（改名映射 / 单位适配器 / 补默认值 / 补枚举分支）；点「应用迁移」下游即恢复正确。
- **不一刀切**：新增可选字段等**安全演进**被识别为向后兼容，静默放行并纳入契约——不是「拦一切改动」。
- 同一套引擎同时驱动「漂移哨兵」的流程回放与「价值可视化」对比条，数字前后一致、随交互实时变化。

## 三个页面

1. **共享盘**：`/drive` 上 3 个工件卡片（生产者 / 消费者 / 契约摘要）+ 生产者→消费者依赖链（纯 CSS）。
2. **漂移哨兵**（核心）：勾选研究员的改动（4 种破坏 + 1 种安全演进，可多选）→ 切换榫卯 OFF/ON →
   OFF 世界展示「静默腐坏爆炸半径」，ON 世界展示「写入时拦截 + 最小迁移 + 下游恢复」，
   底部 before/after 价值条（静默腐坏数 / 检测延迟 / 被污染 agent 数 / 排障回溯步数）。
3. **契约台账**：逐工件契约明细 + 漂移历史时间线 + 契约覆盖率。

## 明确边界（诚实声明）

- **纯前端静态 Demo**：一条 mock 的 3-agent 流水线，**不挂载任何真实文件系统**、
  无登录、不接后端 / 数据库 / 支付 / 外部 API / 真实 LLM。
- 真实产品里「契约」既需**显式声明**也需**自动推断**；agent 输出天然半结构化，
  自动推断 schema 会有噪声，**结构级漂移**（改名/删字段/单位标注）比**语义级漂移**（同名不同义）
  更好判定，后者在真实场景可能需 LLM 辅助推断意图、存在误报——产品需保留「软告警 vs 硬阻断」的分级。
- Demo 里的数字（置信度、爆炸半径、检测延迟…）来自确定性引擎对 mock 流水线的推演，用于说明机制，
  非真实基准测试。
