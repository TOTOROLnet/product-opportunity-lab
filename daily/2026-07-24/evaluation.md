## 7. Demo 的最大问题
机会带一定**前瞻性**：多 agent 共享盘本身尚偏早期（Blaxel 私有预览），当下高痛用户集中在少数重度多 agent 团队；
Demo 用单条 3-agent 流水线说明机制，真实产品的核心工程难点（自动 schema 推断的噪声、语义级漂移判定）未在纯前端体现。

## 8. 是否建议用户人工体验
建议。「漂移哨兵」页的 OFF/ON 一键对比 + 「应用迁移」是最能体现价值的交互，值得亲手点一遍（尤其看 1/100 → 87/100 的结论反转）。

## 9. 下一步建议
- 接真实来源：从编排框架（LangGraph / 多 sandbox）里采集 agent 实际读写的工件，自动推断初版契约；
- 语义级漂移：引入 LLM 辅助判定「同名不同义」，并做「软告警 vs 硬阻断」分级；
- 迁移即代码：把最小迁移导出为可复用适配器（读时/写时），沉淀成团队契约库。
