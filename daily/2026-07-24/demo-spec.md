# Demo 设计说明 — 2026-07-24

## 1. Demo 名称

榫卯 Tenon —— 多 agent 共享工件的「接口契约 + 漂移哨兵」

（榫卯：中国传统木工里两块木头不靠钉子、靠精密咬合的接口。多 agent 的工件也应靠**契约**互相咬合；
一旦一方悄悄变形，榫头就对不上卯眼——这正是我们要在写入时捕捉的「漂移」。）

## 2. 一句话定位

给 Blaxel Agent Drive 这类多 agent 共享盘架一层「类型系统 + 集成测试」：
在生产者 agent 改坏工件形状/单位/语义、下游 agent 静默误读之前，于**写入时**拦截、算清**下游影响**、给出**最小迁移**。

## 3. 目标用户

正在搭多 agent 流水线 / 编排的开发者与 agent 平台团队——多个 agent（或 sandbox）通过共享盘交换中间工件。

## 4. 使用场景

一条 3-agent 研究流水线共用一块 `/drive`：
**研究员** 写 `findings.json` →（分析师读它，写 `metrics.json`）→（报告员读它，写 `report.md`）。
研究员这次「优化」了输出（把 `score` 改名为 `confidence`、把百分比从 `0–100` 改成 `0–1`、删掉 `sampleSize`…），
下游会发生什么？开/关榫卯两种世界对比给你看。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / CLI / API / 基础设施类产品（多 agent 共享文件系统之上的正确性中间层）
- 采用的演示方式：纯前端「**模拟体验 + 价值可视化**」，组合三种形式：
  - **资源/结构视图**：把共享盘的工件、生产者→消费者依赖图、每个工件的契约（schema）可视化；
  - **before / after 对比**：同一次「生产者改动」在「关榫卯 vs 开榫卯」两种世界下的下游结局对比；
  - **模拟工作流回放**：脚本化重放「改动 → 传播/拦截 → 下游结果」的核心流程。
- 演示的是**我们分析出的创新切入点**（工件接口契约 + 写入时漂移哨兵 + 下游影响/迁移），
  **不是** Blaxel 存储层的克隆，也**不是**治理/授权面板。

## 6. 核心流程（用户 3 分钟内可完成）

1. 进「共享盘」页：看到一块 `/drive` 上 3 个工件卡片（生产者 / 消费者 / 契约摘要 / 状态），
   顶部一张生产者→消费者依赖图（纯 CSS，无 SVG）。一眼看懂「谁产谁消费、靠什么契约咬合」。
2. 进「漂移哨兵」页（核心）：从下拉里让**研究员发起一次改动**（预置 4 种典型漂移，可多选）。
   用一个总开关切换 **榫卯 OFF / ON**：
   - **OFF（今天的默认世界）**：改动被静默接受 → 下游分析师/报告员照跑 → 产出**错误但貌似合理**的结果；
     页面高亮「静默腐坏爆炸半径」：错误传播了几步、几个下游 agent 被污染、错误在第几步才暴露（检测延迟）。
   - **ON（榫卯世界）**：哨兵在**写入时**拦截，指出**违反了契约的哪一条**、**影响哪些下游 agent**，
     并给出**最小迁移补丁**（改名映射 / 单位适配器 / 默认值）。点「应用迁移」→ 下游保持正确。
   底部一排 before/after 价值条：静默错误数、检测延迟（步）、被保护的下游工件数、排障步数。
3. 进「契约台账」页：列出每个工件的契约（字段/类型/单位/取值域）、版本与漂移历史（谁/什么时候/改了什么/是否破坏契约），
   展示契约随时间「越用越厚」（覆盖率上升）。

## 7. 页面结构（<= 3 个主要页面）

- 页面 1 **共享盘 Drive**：工件卡 + 依赖图 + 契约摘要（结构/资源视图）。
- 页面 2 **漂移哨兵 Sentinel**：改动选择器 + OFF/ON 开关 + 爆炸半径可视化 + 拦截/迁移 + before/after 价值条（核心交互）。
- 页面 3 **契约台账 Ledger**：逐工件契约明细 + 漂移历史时间线 + 覆盖率增长。

## 8. 关键交互

- **改动多选**：勾选 4 种预置漂移（改名 / 改单位 / 删字段 / 新增枚举）的任意组合，引擎实时重算下游结局。
- **榫卯 OFF/ON 总开关**：同一改动，两种世界即时切换，指标同步变化（同一套引擎驱动，前后一致）。
- **应用最小迁移**：ON 世界里点按钮，把「拦截」变「修复」，下游从「本会被污染」回到「安全」。
- 所有数字（爆炸半径、检测延迟、被保护数、排障步数）均由 `src/logic/engine.ts` 确定性算出，非硬编码文案。

## 9. 使用的 mock 数据

- 一条 mock 的 3-agent 流水线：研究员/分析师/报告员，各自的输入工件、输出工件与读取字段（依赖边）。
- 每个工件一份 mock 契约（字段名、类型、单位、取值域、状态）。
- 4 种预置漂移事件（rename / unit-change / drop-field / new-enum），每种带：是否破坏契约、破坏了哪条、
  典型下游误读后果（如单位从 % 变 0–1 → 分析师把 87 当成 8700%）、对应最小迁移补丁。
- 全部为 mock，**不接**真实文件系统 / 后端 / 数据库 / 外部 API / 真实 LLM。

## 10. 明确不做什么

- 不做登录 / 用户系统
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实 LLM
- 不做生产级后端，不真的挂载/读写任何文件系统
- 不做治理/授权/审批面板（刻意与 valv/Kastra/HOL Guard 家族区隔）
- 不超过 3 个主要页面，数据全 mock

## 11. 成功体验标准

- 用户 3 分钟内能说出：**解决什么**（多 agent 共享盘上工件悄悄漂移导致下游静默误读）、
  **怎么用**（生产者写入时哨兵按契约拦截 + 给下游影响图 + 最小迁移）、
  **增量在哪**（Blaxel 给管道、榫卯给管道里工件的契合保证；治理栈管动作、榫卯管数据结构契约）。
- OFF/ON 切换后，「静默错误数」「检测延迟」「被保护下游数」有直观、可信、前后一致的差异。
- 用户能明确感到这**不是**又一个存储层、也**不是**又一个授权面板。
