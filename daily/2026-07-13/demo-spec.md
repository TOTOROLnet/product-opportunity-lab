# Demo 设计说明 — 2026-07-13

## 1. Demo 名称
**Mneme（记衡）— 跨工具自持记忆的调阅治理层**

## 2. 一句话定位
入库 ≠ 可信，接通 ≠ 全看：在记忆被 `recall` 注入进某个 AI 工具的上下文之前，先按**作用域**过滤（最小权限）、再做**卫生**治理（出处 / 矛盾 / 过期 / 投毒），只放行干净、正确、最小的上下文。

## 3. 目标用户
- 把记忆"自持"起来、接进多个 AI 工具的进阶个人 / 开发者（Second Brain / 未来混元记忆服务的用户）。
- 想给终端用户安全提供"可信记忆"的 Agent / 模型平台方。

## 4. 使用场景
你有一份统一的自持记忆（跨 Claude / Cursor / ChatGPT / 健康 App / 财务 App 写入）。现在一个**新工具接进来并 `recall`**——它究竟应该看到哪些记忆、这些记忆可不可信？Demo 用一个"旅行规划 agent"接入的真实叙事展示裸调阅的风险，以及记衡治理后的结果；再用一个"日程 agent"的 CONTROL 场景证明治理层不逢事就拦。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ **抽象 / API / 基础设施类产品**（记忆调阅治理面，本质是控制面/中间件）
- 采用的演示方式：纯前端「**模拟体验 + 价值可视化**」，核心形式 = **before / after 对比**（裸调阅 vs 记衡治理 TOGGLE）+ 逐条记忆的**出处/裁决卡** + 交互裁决（解决矛盾、确认/退休记忆、切换接入工具看 scope 差异）+ 顶部**信任仪表**。
- 演示的是**我们分析出的创新切入点**（recall→注入之间的作用域 + 卫生治理），不是被参考产品 Second Brain（存储/管道）的克隆。

## 6. 核心流程（用户 3 分钟内可完成）

1. 进入首屏：左侧一份"统一自持记忆"清单（带 scope 标签与出处），顶部一个**接入工具选择器**（旅行 agent / 日程 agent-CONTROL）与一个 **裸调阅 ⇄ 记衡治理** 开关。默认"旅行 agent + 裸调阅"。
2. **看裸调阅的风险**：裸调阅下，`recall` 把**一切**都注入上下文——顶部信任仪表显示"注入 N 条，其中 越界X / 过期Y / 矛盾Z / 未确认W 全部照单注入"，被泄漏/污染的条目用红色角标警示但仍被注入。
3. **打开记衡治理**：同一份 recall，被逐条治理：越界项被拦截（scope 过滤）、过期项标记降权、矛盾项被拎出强制裁决、投毒/未确认项被扣留。信任仪表变绿/黄，显示"实际注入 M 条·拦截 K 条"。
4. **交互裁决**：点矛盾项 → 选保留哪条（另一条退休）；点未确认项 → 确认（升为已核实并放行）或退休；点任一条 → 展开**出处卡**（来源工具/agent、写入时间、用户陈述 vs agent 推断）。裁决后信任仪表实时更新。
5. **切到 CONTROL（日程 agent）**：它读取的都在 scope 内、新鲜、无矛盾、用户陈述 → 记衡显示"**全部放行 · 无需干预 ✓**"，证明治理层不是逢事就拦（反 crying-wolf）。

## 7. 页面结构（<= 3 个主要页面 / 单页分区）
- 页面 1（主）：**记忆治理台** —— 顶部控制条（接入工具选择器 + 裸/治理开关 + 信任仪表）；左栏统一记忆源；右栏"本次注入结果"（逐条裁决卡）。
- 页面 2（抽屉/弹层）：**出处 & 裁决卡** —— 点某条记忆展开：出处、时间、origin、状态、越界/矛盾/过期/投毒原因、可执行动作（裁决矛盾 / 确认 / 退休）。
- 页面 3（说明区）：**这是什么 / 为何非照抄** —— 简述记衡与 Second Brain（存储）的关系，锚定报告信号（放页面底部信息区）。

## 8. 关键交互
- 裸调阅 ⇄ 记衡治理 TOGGLE（核心 before/after）。
- 接入工具切换（旅行 agent / 日程 agent-CONTROL）→ 同一记忆源、不同 scope → 不同注入结果。
- 矛盾裁决（二选一保留）、未确认项确认/退休、过期项退休。
- 点记忆条目展开出处卡。
- 顶部信任仪表随裁决实时联动（已核实 / 推断未确认 / 过期 / 越界 / 已拦截 计数 + 总体横幅）。

## 9. 使用的 mock 数据
全部为 mock，不接真实后端 / 外部 API / 记忆存储。结构（`src/data/scenarios.ts`）：
- `MemoryItem`：`id / subject / content / scope('work'|'personal'|'health'|'finance'|'travel'|'secret') / source{tool,agent?,writtenAt} / origin('user-stated'|'agent-inferred') / status('fresh'|'stale'|'unconfirmed') / contradictsId? / note?`。
- `ConnectingTool`：`id / name / purpose / allowedScopes: scope[]`（旅行 agent 只允许 travel+personal；日程 agent 只允许 personal）。
- 2 个场景：
  - **旅行 agent（风险场景）**：recall 候选含 ①代码库 API key（scope=secret，越界泄漏）②"我住在北京"（stale，已搬家上海）③"偏好靠窗" vs "偏好过道"（矛盾对）④"用户是素食者"（agent-inferred·unconfirmed，某工具写入的幻觉，投毒）⑤合法项若干（travel/personal、fresh、user-stated）。
  - **日程 agent（CONTROL）**：recall 候选都在 personal scope 内、fresh、user-stated、无矛盾 → 记衡全部放行。
- 治理逻辑在 `src/logic/mneme.ts`（`governRecall` 逐条裁决 + `computeSummary` 出总体横幅）。

## 10. 明确不做什么
- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实记忆存储（不复刻 Second Brain 的存储管道）。
- 不做生产级后端、不做真实语义检索 / 真实矛盾检测模型（用 mock 标注展示价值）。
- 不超过 3 个主要分区，全部数据 mock。

## 11. 成功体验标准
- 用户 3 分钟内能说出："它解决什么（跨工具记忆的越界泄漏/投毒/腐烂/出处盲）、怎么用（recall→注入前先 scope 过滤 + 卫生治理）、增量在哪（相对 Second BrainＵ存储，它做注入前治理）"。
- before/after 对比一眼可见：同一份 recall，裸调阅照单全收（含泄漏/投毒），记衡治理后干净、最小、可信。
- CONTROL 场景证明治理层不逢事就拦（无风险即全部放行）。
