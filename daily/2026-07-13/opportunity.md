## 2. 候选机会（至少 3 个，均为创新切入点而非照抄）

### 候选 A：Mneme（记衡）— 跨工具自持记忆的「调阅治理层」
- 创新切入点：不做记忆**存储**，而做记忆在 `recall` 与"注入进某个工具上下文"之间的**治理闸门**。两件事：①**作用域（least-privilege scope）**——按 work/personal/health/finance/travel/secret 给每个接入工具配"可读作用域"，recall 时先按 scope 过滤、越界读取告警；②**卫生（trust at recall）**——对将注入的每条记忆标注**出处**（哪个工具/agent 写入、何时、是"用户陈述事实"还是"agent 推断"）、检测**矛盾**、标记**过期**、拦截**投毒**（幻觉记忆）。
- 目标用户 / 场景：把记忆自持起来的进阶个人 / 开发者（Second Brain / 未来混元记忆服务的用户）；以及要给用户提供"可信记忆"的 Agent 平台方。
- 五维评分：痛点 **4**/5（真实且结构性，但今天跨工具共享记忆尚早、需求部分为前瞻）、机制新意 **5**/5（个人级跨工具记忆的"调阅治理"是全新品类，存储层产品都没做）、方向相关 **5**/5（正中当日 #1 信号：记忆/上下文即基础设施 + MCP）、一天可行性 **5**/5（纯前端 before/after + 交互裁决，全 mock）、启发价值 **4**/5（点出报告漏掉的结构性缺口）→ 合计 **23/25**

### 候选 B：Reality Diff（真形状对账）— OpenAPI 假设 vs 沙箱真形状的差异报告
- 创新切入点：不重做 FetchSandbox 的沙箱本身，而是产出一份**"agent 从规格里假设了什么" vs "有状态沙箱里真实是什么"的差异报告**——字段形状、ID 真伪、异步/webhook 状态迁移、失败模式，作为可复用的集成评审 / 回归物。
- 目标用户 / 场景：写第三方 API 集成的开发者 / coding agent，苦于"mock 通过、上线炸"。
- 五维评分：痛点 **4**/5、机制新意 **3**/5（切入较薄，且逼近 FetchSandbox 本体、有照抄风险）、方向相关 **4**/5、一天可行性 **4**/5、启发价值 **3**/5 → 合计 **18/25**

### 候选 C：Remix（留用）— 把一次性生成小应用沉淀为可复用个人工具箱（2C）
- 创新切入点：JustVibe 每次生成一个"用完即弃"的小应用；缺口是**留存**。做一层把生成的小应用**持久化 + 可复用 + 记住个人偏好**的个人工具箱，差异点是"沉淀而非猎奇"。
- 目标用户 / 场景：常有临时小工具需求的普通消费者。
- 五维评分：痛点 **3**/5（nice-to-have、留存未验证）、机制新意 **3**/5、方向相关 **4**/5（2C AI-native）、一天可行性 **4**/5、启发价值 **3**/5 → 合计 **17/25**

## 3. 最终选择

- 选中：**候选 A — Mneme（记衡）**，总分 **23/25**
- 为什么选它（相对其他候选的优势）：
  1. **正中当日最强信号**：报告明确把"记忆/状态即独立基础设施"列为趋势 1（≥2 产品同向）与"最想跟进方向"，A 直接骑这条趋势，但切入的是报告**没点出**的治理层（不是照抄 Second Brain 的存储）。
  2. **护城河更深**：存储 + 语义 recall 会被厂商内建记忆抹平（报告自己承认"厂商内建更无缝"）；而"注入前的作用域 + 卫生治理"是合并记忆后**新涌现**、且随生态成熟越来越痛的问题。
  3. **B 有照抄风险**（逼近 FetchSandbox 本体）、**C 痛点弱且留存未验证**；A 在机制新意、方向相关、一天可行性上都最高。
  4. **可 3 分钟讲清**：一个"裸调阅 vs 记衡治理"的 before/after，用一个旅行 agent 的真实叙事就能让人当场看懂增量。

## 4. 深入论证（针对选中机会）

1. 目标用户：
   - 主要：已经把记忆"自持"起来的进阶个人与开发者（Second Brain 用户、把一份记忆接进 Claude/Cursor/ChatGPT/健康财务 App 的人）。
   - 次要：想给终端用户提供"可信记忆"的 Agent / 模型平台方（如报告建议的混元记忆服务），他们需要一层治理面才能安全对外暴露记忆。
2. 核心痛点（真实、高频、可付费）：
   - **越界泄漏**：一份记忆接进多个工具后，每个工具都能 recall 全部——代码库密钥、私人健康、财务信息在工具间无差别流动。
   - **投毒扩散**：一个工具的 agent 写入幻觉"事实"，经 recall 注入所有工具，错误被放大成"处处共识"。
   - **记忆腐烂**：过期事实（已搬家、已换工作）与相互矛盾的偏好长期驻留，被反复注入却无人裁决。
   - **出处盲区**：记忆被注入新工具时，用户不知道它从哪来、何时写、是"我说的"还是"某个 agent 猜的"。
   - 这些都是"用完就痛、且随接入工具数量增长而放大"的问题，进阶用户与平台方都有付费意愿（数据安全 + 输出质量）。
3. 现有产品为什么没解决好：
   - Second Brain / 厂商内建记忆都停在**存储 + 检索**层，`recall` 是"语义召回一把梭"，没有作用域、没有出处、没有矛盾/过期/投毒治理。
   - 传统企业数据治理（DLP / 权限系统）面向的是文档与数据库，不是"个人级、跨 AI 工具、注入前动态治理的记忆条目"，粒度与时机都不对。
4. 为什么适合做轻量 Demo：
   - 治理逻辑天然是"输入一组候选记忆 + 一个接入工具 → 输出逐条裁决 + 一个信任摘要"，纯前端可完整表达。
   - "裸调阅 vs 记衡治理"的 before/after 对比，是把抽象基础设施价值可视化的最佳形式（符合 loop 的 CLI/基础设施类演示分型）。

## 5. 不照抄声明（四项必填）

1. 创新切入点是什么：一层站在 `recall` 与"注入进工具上下文"之间的**调阅治理面**——**作用域（least-privilege per-tool scope）** + **卫生（出处 / 矛盾 / 过期 / 投毒治理）**。它**不存储**记忆，只在记忆进入某个工具前对其治理。
2. 解决的具体问题 / 场景：把一份自持记忆接进多个 AI 工具后出现的**越界泄漏、投毒扩散、记忆腐烂、出处盲区**；场景是"一个新工具（如旅行 agent）接进来 recall 时，究竟该看到什么、能不能信"。
3. 相对被参考产品的增量价值（差异点）：Second Brain 解决"记忆存哪、怎么搜、归谁所有"；Mneme 解决"**这条记忆该不该进这个工具、进去之前可不可信**"。前者是存储管道，后者是注入前的治理闸门；前者让 recall"能召回"，后者让 recall"可信、最小、不泄漏"。
4. 为什么这不是照抄 Second Brain（本质区别）：Second Brain 是**记忆存储/管道**（remember/recall/forget + 语义检索 + 自托管）；Mneme **完全不碰存储**，它是叠加在任意此类存储之上的**治理/卫生控制面**，对象是"即将被注入某工具上下文的候选记忆集合"，时机是 recall→注入之间，失败模式是"越界 / 投毒 / 腐烂 / 出处盲"。若拿走 Second Brain 的存储，Mneme 依然能治理任何来源的记忆流；二者是"仓库"与"海关/质检"的关系，不是同一产品。

## 6. 不确定性与风险

- 需求侧风险：跨工具共享记忆今天仍很早（Second Brain 刚起步、自托管门槛高），"注入前治理"的痛点对多数人是**前瞻性**的；若共享记忆没跑起来，这层治理也无用武之地。缓解：先服务"已经痛"的进阶用户与平台方（他们最先撞上泄漏/投毒）。
- 落地 / 技术风险：矛盾检测、出处标注、投毒识别在真实系统里需要可靠的记忆元数据与模型判断；Demo 用 mock 标注展示价值，真实实现的准确率是核心工程难点（会 crying-wolf 或漏判）。
- 我可能判断错的地方：①厂商可能把治理直接内建进各自记忆里，独立治理层的空间被压缩；②"作用域"可能被更简单的"多个独立记忆库"方案替代（用户自己分库），而非一层动态 scope；③进阶用户是否愿意为"注入前治理"付费，需真实验证。
- 为对冲 crying-wolf 风险，Demo 特意加入一个 **CONTROL 场景**（一个日程 agent 读取的都在 scope 内、新鲜、无矛盾、用户陈述）→ 记衡应"全部放行·无需干预"，证明治理层不是逢事就拦。
