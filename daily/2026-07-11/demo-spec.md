# Demo 设计说明 — 2026-07-11

## 1. Demo 名称

**Rootline（根脉）** —— 模型内多智能体调用的「结论溯源 + 压缩失真」审计透镜

## 2. 一句话定位

给一次 `multi_agent` / `ultra` 调用产出的**每一条承重结论**，追一条穿过子代理树与自动压缩层的**证据根脉**，照出被 compaction 悄悄丢掉、却足以推翻结论的关键事实。

## 3. 目标用户

- 正在把编排从应用层迁到 GPT-5.6 `multi_agent` beta / Muse Spark subagent 的 AI 工程团队；
- 把多智能体最终结论直接接下游高风险动作（下单 / 部署 / 赔付 / 对外发布）的产品负责人；
- 需要给「模型自动压缩」加一道信息完整性审计的风控 / 合规。

## 4. 使用场景

一次多智能体调用返回了一个**看起来毫无破绽**的最终答案（如「订 AI-887，¥3,180，落地签可用，在 ¥3,500 政策内」）。它由一棵子代理树聚合而来，每个子代理的上下文都被服务端自动压缩过、只把摘要上报。用户想在**执行动作之前**确认：这条结论背后的关键事实，在层层压缩里到底**活没活下来**。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / API / 基础设施类产品（模型原生多智能体是 API 原语，过程是原子黑盒）
- 采用的演示方式：**模拟体验 + 价值可视化 + before/after 对比**
  - **裸答案 vs 根脉透镜**一键切换（借鉴可视化对比手法）：先看模型给的自信最终答案，再开启透镜看它的证据根脉。
  - **可交互子代理树**：把「单次调用内 spawn 的黑盒树」可视化成可点击节点，每个节点标注角色、token、是否被压缩、压缩丢了什么。
  - **结论 → 根脉溯源**：点最终答案里的某条承重结论，高亮它到源头叶子代理的整条向下路径，并展开每一跳的「压缩 diff」。
- 演示的是**我们分析出的创新切入点**（压缩 + 聚合树的信息存续审计），不是 GPT-5.6 / Muse Spark / Ship OS 的克隆。

## 6. 核心流程（3 分钟内可完成）

1. 进入首屏 → 顶部是一次多智能体调用的**任务 prompt** + 模型给出的**最终答案**（默认「裸答案」态，看起来很自信、可直接执行）。
2. 点「开启根脉透镜」切换 → 中央出现**子代理树**（根 → 子代理 → 叶子），被压缩过的节点带 ⚠ 压缩徽标；右侧最终答案的每条承重结论出现**存续 / 失真 / 断链**色标。
3. 点任意一条结论（如价格 ¥3,180）→ 树上高亮该结论的**根脉路径**，右侧展开每一跳的「原始发现 vs 压缩后上报」diff，红色标出**被丢弃的承重事实**（如「+¥900 行李费」）。
4. 顶部**真相计**汇总：X 条承重结论中 Y 条被压缩降级、Z 条影响可执行决策 → 给「可否照此执行」的明确判断。
5. 切换场景（旅行预订 / 运维根因 / 文献综述-对照组）验证：对照组里所有事实都存续，透镜**全绿背书**——证明它不是「逢结论必报警」。

## 7. 页面结构（<= 3 个主要页面）

- 页面 1（主页面，单页三栏 / 响应式）：
  - 顶栏：场景切换器 + 裸答案/透镜 切换 + 真相计。
  - 左 / 中：子代理树（可点击、压缩徽标、根脉高亮）。
  - 右：多智能体最终答案 + 每条承重结论色标 + 选中结论的根脉溯源面板（逐跳压缩 diff）。
- 页面 2：「它如何工作 / 方法论」折叠区（解释根脉、compaction 损失、三种判定，并声明全为 mock）。
- 页面 3：无（控制在 1 个主页面 + 1 个说明区，避免过度）。

## 8. 关键交互

- **裸答案 ↔ 根脉透镜** 切换（核心 before/after）。
- 点树节点 → 展开该子代理的「原始发现 / 压缩后上报 / 被丢弃事实 / token」。
- 点最终答案里的结论 → 高亮根脉路径 + 打开逐跳压缩 diff 弹层。
- 场景切换 → 三套 mock（含一套全存续对照组）。
- 结论色标：**存续（绿）/ 失真（琥珀）/ 断链（红）**；对影响可执行决策的结论额外打「⛔ 影响执行」标记。

## 9. 使用的 mock 数据

- 结构（`src/data/scenarios.ts`，全部为**带标注的 mock**，UI 与 README 明示）：
  - `AgentNode`：`{ id, role, parentId, tokensIn, tokensOut, compacted, rawFindings[], reportedUp, droppedFacts[] }`——`rawFindings` 是叶子拿到的原始事实，`reportedUp` 是压缩后向上汇报的摘要，`droppedFacts` 是压缩中被丢弃 / 弱化的承重事实。
  - `Claim`：最终答案里的一条承重结论 `{ id, text, sourceLeafId, survival: 'intact'|'degraded'|'broken', affectsAction, explanation }`。
  - `Scenario`：`{ id, title, prompt, finalAnswer, nodes[], claims[] }`。
- 三套场景：
  1. **旅行预订**：最终「¥3,180、落地签、在政策内」；实际叶子发现「¥3,180 不含行李，+¥900」「落地签仅限停留 <15 天，而会期 18 天」在压缩中被丢 → 2 条失真且影响执行、总价其实破 ¥3,500 政策。
  2. **运维根因（RCA）**：最终「根因＝配置 X 变更，可安全重部署」；日志子代理上下文被压缩，丢掉「同一报错在配置 X 变更前已出现」这一反证 → 根因结论**断链**（根代理据摘要自行合成了因果）。
  3. **文献综述（对照组）**：所有叶子关键事实都存续，透镜判定全部**存续（绿）**并背书结论——反「狼来了」。
- 逻辑（`src/logic/rootline.ts`）：`getPath(nodes, leafId)`（叶子到根的路径）、`classifyClaim(claim)`、`summarize(scenario)`（存续/失真/断链计数 + 是否可执行的整体判断）。所有判定来自 mock 标注，无真实模型 / 后端 / 外部 API。

## 10. 明确不做什么

- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实模型 API（不调用 OpenAI / Meta）。
- 不做生产级后端，不做真实 trace 采集。
- 不复刻编排 / 工作台（不做 Ship OS / Sim 的看板派活）。
- 不承诺真实 benchmark / 成本数字（成本不是本 Demo 主张）。

## 11. 成功体验标准

- 用户 3 分钟内能说出：**它解决什么**（多智能体黑盒 + 有损压缩会让自信结论丢掉承重事实）、**怎么用**（对一次调用的每条结论做根脉溯源）、**增量在哪**（裸答案看不出破绽，透镜把「压缩吞掉的破绽」逐条照出，并判断可否执行）。
- 对照组（文献综述）能让用户直观看到「透镜不会逢结论必报警」，从而建立对工具判断力的信任。
