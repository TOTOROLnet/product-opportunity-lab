# Demo 设计说明 — 2026-07-21

## 1. Demo 名称

**信任阶梯 TrustLadder — Agent 写权限的渐进授权层**

## 2. 一句话定位

把你对 agent 动作的「批准/拒绝」决策，实时编译成**可撤销的分级自动放行策略**——让重复的审批被机器接管，人的注意力只收敛到真正低可逆、高风险、新型的动作上；信任一格一格挣，被辜负时一格一格掉。

## 3. 目标用户

同时部署了多个自主 ops agent（应收 / 运营 / 工程）的团队负责人：既被审批队列淹没，又不敢把「写权限」整个放开。

## 4. 使用场景

一个上午，三个 agent 持续把「待执行的写动作」推到你面前（发催收邮件、改 CRM、退款、合 PR……）。你在审阅台逐条处置；系统在背后学习你的偏好，主动建议把「你已一致批准过很多次」的低风险类目**提升为自动放行**；提升后同类动作自动流过，你的审批负荷肉眼可见地下降；直到一条被自动放行的动作**出事被反转**——系统立刻把该类信任**降一格**、暂停自动、把它重新交回你手上。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / CLI / API / 基础设施类产品（一层横切 agent 的**治理控制面**，无可视化实体）
- 采用的演示方式：纯前端「**模拟体验 + 价值可视化**」，融合三种形式——
  - **模拟审阅台**（脚本化重放一条多 agent 动作流，用户真实处置）；
  - **资源/结构视图**（把抽象的「信任」可视化成每个类目的**信任阶梯（0–5 格）**与其升降历史）；
  - **before / after 对比**（同一条脚本，用「全人工闸门」与「信任阶梯」两种策略跑，实测审批负荷、覆盖率、关键动作漏审数）。
- 演示的是**我们分析出的创新切入点**（从审批行为归纳出可撤销的分级授权 + 信任状态机），**不是** Rex/Nautis/Lunen 的克隆。

## 6. 核心流程（3 分钟内可完成）

1. 进入 **审阅台**：看到一列待审的 agent 动作，每条带「所属 agent / 人话摘要 / 类目 / 可逆性徽章 / 金额-风险 / 该类目当前信任阶」。
2. 逐条处置（批准 / 拒绝 / 改后批准 / 升级给人）。处置同一类目累积「一致证据」。
3. 当某类目证据达标，顶部弹出**「提升为自动放行？」**建议卡（附人话规则 + 证据统计）。点「提升」→ 该类目爬升一阶。
4. 继续放行：后续**同类动作自动流过**（不再拦你），右侧计数器「需人工审阅数 / 自动放行覆盖率」实时变化。
5. 脚本进行到**反转事件**：一条此前自动放行的动作被标记「客户投诉/被反转」。系统即时把该类目**信任降一格**、暂停自动、把后续同类动作退回人工，并在审计里记明原因。
6. 打开 **价值对比** 页：同一条脚本下「全人工闸门 vs 信任阶梯」的实测指标并排，看清「省了多少人工、且关键动作零漏审」。

## 7. 页面结构（<= 3 个主要页面，用顶部 Tab 切换）

- 页面 1：**审阅台（Cockpit）**——待审队列 + 逐条处置 + 自动放行建议卡 + 右侧实时计数器 + 反转事件横幅。
- 页面 2：**信任阶梯（Ladder）**——所有类目的信任阶（0–5 格）总览、每格的证据/升降历史、当前生效的**可撤销策略**列表（可一键「收回自动」）。
- 页面 3：**价值对比（Compare）**——「全人工闸门」基线 vs「信任阶梯」策略，在同一脚本上的实测指标 + 差异说明 + 安全性红线（高风险/不可逆动作漏审数必须为 0）。

## 8. 关键交互

- 审阅台逐条：批准 / 拒绝 / 改后批准 / 升级；处置后动作从队列移除并进入审计流。
- 建议卡：达阈值时出现，展示「规则 + 证据 n/n + 影响范围」，可「提升」或「暂不」。
- 信任阶梯：每类目一条 0–5 格进度轨；hover/点开看升降历史与生效规则；「收回自动」一键降级。
- 反转横幅：脚本触发，红色高亮，点击定位到被降级的类目。
- 播放控制：审阅台支持「自动播进下一条 / 单步」，让脚本化动作流按节奏到达。
- 对比页：一个「重放本脚本」按钮，确定性地重算两种策略的指标（不依赖用户手速，保证结论稳定）。

## 9. 使用的 mock 数据

全部为 mock、脚本化、确定性，**不接**任何真实后端/agent/LLM/外部 API：

- `src/data/scenario.ts`：一条约 20+ 条动作的脚本化「一个上午」，字段含 `id / tick / agent(应收|运营|工程) / category / summary(人话) / reversibility(high|medium|low) / stakes(金额或影响) / groundTruth(该动作事后是否会出问题) / isReversalOf(标记反转)`。
- `src/logic/engine.ts`：确定性引擎——
  - 信任阶梯状态机：每类目 rung 0–5，连续一致批准 → 到阈值可提升；提升后同类自动放行；命中 `groundTruth=bad` 的自动放行 → 立即降一格 + 暂停自动。
  - 两种策略回放：`manual`（全人工，每条都要人审）与 `trustladder`，在同一脚本上算出：人工审阅次数、自动放行覆盖率、高风险/不可逆漏审数、被反转数、清空队列的「模拟耗时」。
- 类目示例：`应收提醒·小额`、`CRM 字段更新·低风险`、`退款·大额`、`信用冻结`、`合并 PR·非核心`、`法务威胁升级` 等；其中大额/不可逆类目设计为**永远不该被自动放行**，用于验证安全红线。

## 10. 明确不做什么

- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实 LLM / 真实 agent。
- 不做生产级后端；不做真实的可逆性估计模型（用确定性代理）。
- 不做真实 MCP / ERP / 邮件系统对接（仅以 mock 摘要呈现动作）。
- 页面不超过 3 个；全部数据 mock。

## 11. 成功体验标准

- 用户 3 分钟内能说出：**它解决什么**（审批疲劳 vs 过度放权的两难）、**怎么用**（处置 → 系统建议提升 → 自动放行 → 反转即降级）、**增量在哪**（从行为学出来的、可撤销的分级授权，而非事前写死或只写不读的审计）。
- 用户能亲手把一个类目「养」到自动放行，并亲历一次反转把信任打回去。
- 价值对比页给出可信的 before/after：审批负荷显著下降，且**高风险/不可逆动作漏审数 = 0**（证明「省力不等于失控」）。
