## 2. 候选机会（至少 3 个，均为创新切入点而非照抄）

### 候选 A：屏见（ScreenConsent）— 桌面悬浮 Agent 的"截屏前脱敏 + 可见范围同意层"
- 创新切入点：在 Claude Overlay 这类"把整屏喂给模型"的悬浮 agent 与模型之间，插一层**发送前预览**：自动识别并遮罩敏感区域（密码框、token、其他 App 窗口、私人聊天），用框选限定"agent 这次到底能看到哪块"，给出"模型将看到的画面"before/after 预览与逐次同意留痕。
- 目标用户 / 场景：用悬浮编码 agent 但担心整屏泄露/误改的开发者与团队。
- 五维评分：痛点 4/5，机制新意 4/5，AI核心度 3/5，一天可行性 4/5，启发价值 4/5 → 合计 **19/25**
- 备注：真实且切题，但**本质仍是"控制/同意面"**——与我 07-03 Fusebox、07-12 明约同族，对本实验室边际启发有限，今天想避免第 N 次做控制台。

### 候选 B：列队（Standup）— "AI 员工团队"的经理可读晨会视图 + 待你决策收件箱
- 创新切入点：面对 ClawTeams/Pazi 那种"一群 agent 在跑"的火水管日志，做一层**经理视角**：把 30+ agent 动作折叠成"组织架构 + 每人目标/阻塞 + 关键路径 + 只给你留一个『待决策收件箱』"。
- 目标用户 / 场景：把目标委托给多 agent 团队、但看不懂/看不过来 Live View 的管理者。
- 五维评分：痛点 4/5，机制新意 3/5，AI核心度 3/5，一天可行性 4/5，启发价值 4/5 → 合计 **18/25**
- 备注：Live View 已存在，"晨会化"是增量改良而非新范式；且又是"监督/编排可视化"家族，仍偏科技术向控制面。

### 候选 C：耳记（Earmark）— 把任意文档变成"边听边被追问"的主动回忆音频课
- 创新切入点：不做"又一个能听的播客"，而是把"文档→音频"从**内容消费**重构为**主动回忆式学习**。AI 把材料切成概念单元，为每个单元生成音频讲解**+苏格拉底式追问**；播放到概念边界会**自动暂停发问**，你答完立刻反馈，系统据此建**保留度地图**，把没记住的概念排进"60 秒重听"队列。差异不是"脚本可控"，而是**"听完真的记住"**。
- 目标用户 / 场景：要把长报告/教材/论文"听掉"的学生、自学者、知识工作者——通勤/碎片时间既想省眼睛又想真学会。
- 五维评分：痛点 4/5，机制新意 5/5，AI核心度 4/5，一天可行性 5/5，启发价值 4/5 → 合计 **22/25**

## 3. 最终选择

- 选中：**候选 C — 耳记（Earmark）**，总分 **22/25**
- 为什么选它（相对其他候选的优势）：
  1. **跳出偏科**：A、B 都是"控制/监督面"，本实验室近 12 天几乎全是这类；C 是 AI-native 的 2C 学习体验，正面回应 `lab-focus` "不要系统性忽略 2C"。
  2. **重构品类而非改良**：A/B 是给现有形态加控制/视图（增量改良），C 把"文档→音频"的 JTBD 从"消费"改写成"留存"，天花板不同。
  3. **纯前端可真体验**：核心闭环（听→被追问→答→保留度地图→重听队列）完全能用 mock 语料 + 浏览器原生 `speechSynthesis`（非外部 API、无需 key）做成可交互 Demo，一天可行性最高。
  4. **对报告结论有实质修正**：报告给 VocalVia 判低分，我认同其"作为播客生成器"的判断，但指出它错配了 JTBD——这是独立判断而非跟随。

## 4. 深入论证（针对选中机会）

1. 目标用户：需要"消化长文本"的学生、考证/自学者、要吃行业报告与论文的知识工作者；他们已经在用 TTS/NotebookLM"听材料"，但**听完等于没听**。
2. 核心痛点（真实、高频、可付费）：被动听（播客/TTS）是线性、无反馈的，**留存率极低**（教育学上被动听讲的长期留存远低于主动回忆/自测）；用户投入了通勤时间，却说不清自己到底记住了什么、哪里是薄弱点。这个痛点高频（几乎每天）、可付费（教育/效率历来愿付费）。
3. 现有产品为什么没解决好：VocalVia / NotebookLM / Vois 都在优化**"产出更好听的音频"**（消费侧指标：像不像对话、能不能改脚本），没人优化**"听完的留存"**（学习侧指标）。它们把音频当终点，我把音频当**主动回忆的载体**。
4. 为什么适合做轻量 Demo：核心价值靠**交互闭环**说清，不依赖真实 TTS 音质或真实模型——用 mock 语料（预切好的概念单元 + 预生成的追问）+ 浏览器 `speechSynthesis` 就能让人 3 分钟内体验"听→被打断追问→看到自己的保留度地图→拿到重听队列"，并用 before/after 直观对比被动听 vs 主动回忆的估算留存。

## 5. 不照抄声明（四项必填）

1. **创新切入点是什么**：把"文档→多角色音频"从"内容消费产品"重构为"**主动回忆式学习产品**"——音频不是终点，而是触发主动回忆（active recall）的载体：边听边被苏格拉底式追问，实时生成个人保留度地图，并自适应地把薄弱概念排进短时重听。
2. **解决的具体问题 / 场景**：用户花通勤/碎片时间"听掉"长材料，却**留存极低、说不清薄弱点**。耳记把这段被动收听时间转化为可测量、可巩固的学习成果。
3. **相对被参考产品的增量价值（差异点）**：VocalVia 的可控性用在"合成前改脚本"（消费侧）；耳记把"花前确认"用在**"我记住了没有"**（学习侧）。核心指标从"音频好不好听"变成"**听完的留存与薄弱点定位**"，并新增了被动听 vs 主动回忆的留存对比、自适应重听队列——这些都是 VocalVia/NotebookLM 没有的学习闭环。
4. **为什么这不是照抄 VocalVia / NotebookLM**：它们的产品终点是"生成一段可听/可改的音频"；耳记的产品终点是"**用户对材料的可测量掌握**"。同样从文档出发、同样产出音频，但目标函数、核心交互（被打断的主动回忆）、留存指标、重听调度完全不同——用户体验耳记不是为了"听个播客"，而是为了"这次真的学会"。

## 6. 不确定性与风险

- 需求侧风险：主动回忆意味着"打断沉浸"，可能与"我就想放松地听"的用户预期冲突——需要"专注模式/背景模式"可切换；愿为"学得会"付费的用户占比需验证。
- 落地 / 技术风险：真实产品里"从任意文档自动切概念单元 + 生成高质量苏格拉底追问 + 给自由作答评分"依赖 LLM 质量，且自然音质是硬门槛；Demo 用 mock 语料回避，但真实上线需解决生成质量与延迟。
- 我可能判断错的地方：也许多数"听文档"用户本就只求"泛听混个耳熟"、并不想被测；若如此，主动回忆是伪需求的可能存在，需要用留存/复购数据证伪，而非我一厢情愿。
