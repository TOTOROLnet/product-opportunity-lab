# Demo 设计说明 — 2026-07-12

## 1. Demo 名称
**明约 Mandate** — 自主办公 agent 的「授权对账」透镜

## 2. 一句话定位
把你**批准过的那份计划**当成一份有约束力的契约，在 agent 自主跑完数小时后，逐步对账「实际执行 vs 你签字的计划」，照出每一处沉默偏离，并在**交付前**用人工签署放行越界动作。

## 3. 目标用户
用 ChatGPT Work / Codex / Notion Ship OS 等自主 agent 产出成品的**非工程业务负责人**（营销 / 运营 / 财务 / 项目 owner）——批准一份计划后离开，数小时后收到成品，需要快速判断「能不能直接用 / 发出去」。

## 4. 使用场景
一次自主运行结束、agent 交付了一份光鲜成品（出海营销简报 / 季度费用报表 / 周会纪要）。用户打开明约，一眼看到「这次运行是否仍落在我批准的边界内」，把越界项签署或驳回后，再决定是否采用。

## 5. Demo 策略分型（重要）
- 本机会属于：☑ 抽象 / 控制面 / 问责层类产品（agent 执行之上的对账透镜，非可视化终端产品）
- 采用的演示方式：纯前端「**模拟体验 + 价值可视化**」，形式为 **before/after 对比 toggle**（裸交付 vs 明约透镜）+ **计划↔执行逐步对账视图** + **成品单元格反查计划步** + **越界项人工签署**。
- 演示的是**我们分析出的创新切入点**（执行后、以计划为锚的反向对账 + 交付口 gating），不是 ChatGPT Work 的克隆——本 Demo **不执行任何办公任务、不产出成品**，只对既有的「计划 / 执行 / 成品」三件套做对账。

## 6. 核心流程
（用户从进入到"看懂价值"的主路径，3 分钟内可完成）
1. 进入即选中一个场景（默认「出海营销 campaign brief」）。左侧是 agent 交付的**成品**，看起来已完成、可直接发出。
2. 打开 **明约透镜 toggle**：右侧出现「授权契约对账」——把批准计划的每一步与实际执行并排，标注 ✅守约 / 🟡微调 / 🔴越界，并给出整体裁定条（⛔需重新签署 / ⚠建议复核 / ✓完全守约）。
3. 点成品里的某个论断 / 单元格：高亮它由**哪一条计划步 + 哪个（mock）数据源**产生，以及那一步是守约还是越界——把「成品」反查回「授权边界」。
4. 对每个 🔴越界项点「签署放行」或「驳回重做」：裁定条与「可否直接交付」实时更新，把「让资深判断留在回路里」变成可点的动作。
5. 切换到「季度费用报表」（未授权数据源）与「周会纪要」（对照组，完全守约、透镜背书可直接用），验证透镜**不虚警**。

## 7. 页面结构（<= 3 个主要页面 / 视图，单页应用内切换）
- 视图 1（主）：**对账工作台** — 顶部场景切换 + 裸/透镜 toggle；左「交付成品」，右「授权契约对账」时间线。
- 视图 2：**单元格反查抽屉** — 点成品元素后从右侧滑出，显示其来源计划步、数据源、守约状态。
- 视图 3：**运行摘要卡** — 顶部一条：守约/微调/越界计数 + 「可否直接交付」裁定 + 未签署越界项数。

## 8. 关键交互
- **裸交付 ↔ 明约透镜** toggle：直观 before/after，透镜关闭时只见光鲜成品（"看着像做完了，信我"）。
- **计划步 ↔ 执行** 逐行对账：点任一步展开「你批准了什么 / agent 实际做了什么 / 差在哪」。
- **成品元素点击反查**：成品里的数字 / 论断高亮其计划步来源与守约状态。
- **越界项签署 / 驳回**：改变整体裁定与可交付状态（人工回路）。
- **场景切换**：3 个标注场景，含 1 个对照组（证明不虚警）。

## 9. 使用的 mock 数据
全部为 mock，不接真实后端 / 外部 API。数据结构（`src/data/scenarios.ts`）：
- `Scenario`：id、标题、成品类型（brief / spreadsheet / doc）、`deliverable`（成品的结构化内容，含可点击元素及其 `sourceStepId`/`sourceRef`）、`planSteps`（批准计划：intent、约束/guardrail）、`executionTrace`（实际执行：每步 action、用到的数据源、是否对外动作）、每步对账 `verdict`（honored/adapted/deviated）、`deviationReason`。
- 3 个标注场景：
  1. **出海营销 campaign brief**：批准=3 市场 / 邮件仅拟稿 / 用批准的 CRM 段 / 预算上限 ¥50k；执行=扩到 5 市场（🔴范围扩张）、真发了一封 warm-up 邮件（🔴未授权对外动作）、换了同级但不同的数据源（🟡微调）→ ⛔需重新签署。
  2. **季度费用报表**：批准=仅用 Drive 文件夹 A + 对账 CRM；执行=有一格数字取自 Slack 估算（🔴未授权数据源），其余守约 → ⚠/⛔；点该单元格可见其来源为「Slack 估算」而非批准账本。
  3. **周会纪要整理（对照组）**：批准=读文档、总结、产出文档；执行=完全照做、无对外动作、仅用批准源 → ✓完全守约，透镜明确背书「可直接采用」（反虚警）。

## 10. 明确不做什么
- 不做登录 / 用户系统
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实 agent 运行时
- 不做生产级后端
- 不真正执行任何办公任务、不产出真实成品（只对既有 mock 三件套做对账）
- 不臆造 token / 成本数字（成本不是本 Demo 主线）

## 11. 成功体验标准
- 用户 3 分钟内能说出：明约**解决什么**（自主 agent 跑完后无法确认是否偏离批准计划）、**怎么用**（开透镜 → 看对账 → 反查成品 → 签署越界）、**增量在哪**（把前向审批补成执行后反向对账 + 交付口 gating）。
- 透镜在对照组场景**明确背书**「可直接采用」，证明它不是逢事必报的「狼来了」。
- 越界项签署 / 驳回能实时改变「可否直接交付」裁定，让「人在回路」可感知。
