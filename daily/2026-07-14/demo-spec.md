# Demo 设计说明 — 2026-07-14

## 1. Demo 名称

**值当（Worthwhile）— 自主 agent 花费策略 + 对账台**

## 2. 一句话定位

在"价格已知"与"实际付费调用"之间加一层：**花前**帮 agent 在一堆带价格的付费源里选"最便宜且够用 + 去重 + 够用即止"，**花后**给出逐笔价值归因的可解释账单，让 agent 花得省、花得值、账算得清。

## 3. 目标用户

- 主要：在 x402 / AgentKey-credit 生态里跑**自主付费 agent** 的开发者与小团队（研究/调研、数据聚合、运营 agent），直接为调用账单买单。
- 次要：想给用户"预算约束地安全花钱"的 Agent 平台方 / 本地 harness（如 Osaurus）。

## 4. 使用场景

一个研究 agent 接了 6~8 个**付费**数据源（每次调用有明确价格），要在预算内产出一份"够用"的简报。
- 现状（贪婪 agent）：能查就查、一把梭，重复来源不去重，够用了也不停，账单是一坨"$X / N 次调用"的黑账。
- 值当介入：花前做取舍与规划、够用即止；花后给出逐笔"值 / 冗余 / 浪费"的归因收据与"相对贪婪基线省了多少"。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / CLI / API / 基础设施类产品（agent 花费策略 + 对账，是一层基础设施能力）
- 采用的演示方式：纯前端"**模拟体验 + 价值可视化**"——以 **before/after 对比**（贪婪基线 vs 值当治理）为主轴，叠加**可交互规划器**（拖动预算与"够用标准"实时改变计划）与**逐笔价值归因收据**。
- **关键诚实声明**：每个付费源的价格 / 覆盖的信息需求 / 覆盖质量 / 重叠关系是**手工 mock 标注**的；但"按边际价值/成本排序 → 去重 → 够用即止 / 预算封顶"的**规划算法是前端真实运行**的，拖动滑块会真实改变计划与账单——是"mock 数据 + 真实优化逻辑"，不是录像式假演示。
- 演示的是**我们分析出的创新切入点**（花前取舍 + 花后对账），不是 AgentKey（接入网关）或 Loomal（支付轨道）的克隆。

## 6. 核心流程（3 分钟内看懂）

1. 进入首屏：选一个任务场景（如"竞品定价与口碑调研"），看到该任务拆出的**信息需求（sub-needs）**与一排**带价格的候选付费源**（价格 + 覆盖哪些需求 + 覆盖质量），并可拖动**预算**与**够用标准**两个旋钮。
2. 点击"运行对比"：左右并排跑 **贪婪 agent（BEFORE）** 与 **值当（AFTER）**，可**逐步播放**决策：值当在每一步解释"这个源 $0.40 与已调 X 重叠 80% → 跳过""这个源 $0.05 最便宜且覆盖'定价'需求 → 调用""信息需求已满足 且 边际价值<成本 → 停"。顶部两条**实时花费计**对比。
3. 看**对账收据**：逐笔列出成本 → 贡献（新信息/重复/低价值）→ 裁决（值 ✓ / 冗余 ⟳ / 浪费 ✗），底部给出结论："同一任务，$1.65，省 $2.55（61%），质量对等（需求覆盖 5/5）"。切换 CONTROL 场景验证：值当在已接近最优的任务上给出"无显著可省"。

## 7. 页面结构（<= 3 个主要页面，采用单页 + 3 视图切换）

- 视图 1 · **任务与预算台（Setup）**：场景选择器、任务与信息需求、候选付费源市场（价格/覆盖/质量）、预算滑块、够用标准滑块、"运行对比"按钮。
- 视图 2 · **对比运行（Compare）**：贪婪 vs 值当 并排时间线，逐步播放决策与理由，实时累计花费与需求覆盖进度条。
- 视图 3 · **对账收据（Receipt）**：逐笔价值归因表 + 裁决 + 省额总结 + "价值归因如何算"的方法说明；CONTROL 场景在此体现"不逢事就喊省"。

## 8. 关键交互

- **场景切换**：3 个 mock 场景（含 1 个 CONTROL 场景）。
- **预算滑块 / 够用标准滑块**：实时重算值当计划——预算越高越舍得多调、够用标准越高越不轻易停手；直观展示"策略随约束变化"。
- **逐步播放 / 一键跑完**：播放值当的每步取舍理由。
- **逐笔收据展开**：点开每笔调用看"为什么判它值/冗余/浪费"。

## 9. 使用的 mock 数据

- 3 个任务场景，每个含：任务描述、若干**信息需求（sub-need）**及权重；一组**候选付费源**，每源含 `name / category / priceUSD / covers(sub-need→质量0..1) / overlapsWith / reliability`。
- 贪婪基线 = 近似"全调用 / 高价优先"的朴素策略；值当计划 = 前端**真实运行**的规划器（边际价值/成本排序 + 去重 + 够用即止 + 预算封顶）。
- 价值归因裁决（值/冗余/浪费）由规划器对每笔调用的边际贡献计算得出。
- 全部为 mock，不接真实后端 / 外部 API / 支付。

## 10. 明确不做什么

- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实 API / x402 结算。
- 不做真实的价值估计模型（价值/覆盖为 mock 标注，仅规划算法真实）。
- 不做生产级后端；不超过 3 个主要视图；全部数据 mock。

## 11. 成功体验标准

- 用户 3 分钟内能说出："它解决什么问题（agent 贪婪花费 + 黑账）、怎么用（选任务→设预算/够用→跑对比→看收据）、增量在哪（不是支付/接入，而是花前取舍 + 花后可解释对账，同任务省 XX% 且质量不降）"。
- 拖动预算/够用标准时计划与账单**实时且合理地**变化（体现真实优化逻辑）。
- CONTROL 场景能给出"无显著可省"，证明不是无脑喊省钱。
