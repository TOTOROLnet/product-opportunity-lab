# Demo 设计说明 — 2026-07-10

## 1. Demo 名称
**Pane（明窗）— 生成式 UI 的可信 / 可读层**

## 2. 一句话定位
给「模型现场生成的界面」加一层玻璃：让每个生成的数字、按钮、对比表都能被追问「这是真的吗？点了会发生什么？这界面几成是编的？」

## 3. 目标用户
- 直接用户：构建生成式 UI 消费产品的团队 / 要提供「生成式界面」能力的平台（如混元）。
- 最终受益者：在生成式 UI 里做「真花钱 / 真操作」决策的普通消费者。

## 4. 使用场景
消费者向一个生成式 UI 助手提问（订机票 / 分析账单 / 换算食谱），助手**现场生成一整套可交互界面**。问题是：界面里的价格、余额、按钮，有多少是真实核实过的，有多少是模型编的、有多少按钮根本没绑定真实能力？Pane 就在这一刻介入。

## 5. Demo 策略分型（重要）

- 本机会属于：**抽象 / 基础设施类产品**（一个叠加在生成式 UI 之上的信任中间层），同时**天然高度可视化**（对象就是界面）。
- 采用的演示方式：**「模拟体验 + 价值可视化」的 before / after 对照**。
  - 用一个 mock 的「生成式 UI 助手」现场生成可交互卡片；
  - 顶部一个开关在 **「裸生成式 UI（Monogram 式，无可信层）」** 与 **「接入 Pane 明窗后」** 之间切换；
  - 裸模式：界面精美、一切看起来都权威可信，无法分辨真伪，点「立即预订」直接执行（危险）；
  - Pane 模式：每个元素带溯源徽标（已核实 / 模型推测 / 占位）、可点开看来源、危险动作被拦截为「明文预检 + 人工确认」、并给出界面级「真实度」仪表。
- 演示的是**我们分析出的创新切入点**（生成式 UI 的信任层），不是 Monogram 的克隆（我们不比谁生成的界面更好，只让生成的界面变得可信）。
- ⚠️ 所有「模型生成」「grounding 判定」均为**预先标注的 mock**，UI 内与 README 均明确标注（诚实性）。

## 6. 核心流程
（用户从进入到「看懂价值」的主路径，3 分钟内可完成）
1. 首屏一句话讲清：生成式 UI 让幻觉穿上了权威 UI 的外衣——你怎么知道这个价格 / 这个按钮是真的？
2. 选择一个场景（默认「订最便宜的东京机票」）→ 看到助手**现场生成**的可交互卡片（比价表 + 「立即预订 ¥1,280」按钮）。
3. 在 **裸 / Pane** 之间切换：
   - 裸模式：卡片精美，无任何提示；点「立即预订」直接弹出「已下单」——用户根本没机会发现那个价格是模型编的。
   - Pane 模式：同一张卡片被打上徽标——某个价格标为「模型推测·未核实」，那个「立即预订」按钮被识别为「未绑定真实预订能力的占位按钮」；点击被拦截为明文预检。
4. 点任意带徽标的元素 → 弹出溯源卡（来源 / 抓取时间 / 为什么被判为推测）。
5. 顶部「真实度」仪表实时显示：本界面 X% 已核实 / Y% 模型推测 / Z% 占位。
6. 切换第二 / 第三个场景（账单分析里抓出一个被编造的「本月支出」总额；食谱换算里展示一个完全 grounded 的健康界面作对照），强化「Pane 在不同界面里都能拆穿或背书」。

## 7. 页面结构（<= 3 个主要页面）
- 页面 1 · **对照体验台（主页面）**：场景切换器 + 裸/Pane 开关 + 生成式界面画布 + 元素溯源 + 真实度仪表 + 动作预检弹层。（核心，承载 90% 价值）
- 页面 2 · **原理说明（「它是怎么判断的」）**：用图文说明 Pane 如何为每个元素判定 grounding（已核实 / 推测 / 占位）与如何拦截动作；明确标注 Demo 中判定为 mock。
- （不做第 3 个独立页面，控制复杂度；首屏价值说明内联在页面 1 顶部。）

## 8. 关键交互
- **模式开关**（裸 vs Pane）：同一份生成界面，一键对照，是全 Demo 的「顿悟时刻」。
- **元素溯源点击**：点带徽标元素 → 溯源弹层（来源、抓取时间、判定理由、可信/不可信）。
- **动作预检拦截**：Pane 模式下点危险/占位按钮 → 弹出明文「此按钮实际会做什么 / 此按钮未绑定真实能力」+ 需显式确认；裸模式下点同一按钮 → 直接「执行」（对比出危险）。
- **真实度仪表**：随场景与模式切换实时更新的分段条（绿/黄/灰）。

## 9. 使用的 mock 数据
- 3 个场景，每个是一份「生成式 UI 描述」（JSON-like 结构）：由若干**带 grounding 标注的元素**组成（类型：数字/文本/对比表行/动作按钮；grounding：verified / guessed / placeholder；附 source、fetchedAt、reason、以及 action 按钮的 realBinding 真伪与明文效果）。
  - 场景 1 · 订东京机票：比价表 3 行（2 行 verified 来自「示例航司数据」、1 个价格 guessed），一个「立即预订」按钮被标为 placeholder（未绑定真实预订能力）。
  - 场景 2 · 本月账单分析：一个「本月支出 ¥8,640」总额被标为 guessed（模型把几笔凑错），其余明细 verified；「一键分期」按钮为真实动作（realBinding=true，明文预检会花钱）。
  - 场景 3 · 食谱按 6 人份换算：全部 verified（对照组——证明 Pane 不是只会报警，也会为可信界面**背书**，避免「狼来了」）。
- 数据结构与「判定逻辑」在 `src/logic/grounding.ts`；场景在 `src/data/scenarios.ts`。全部为 mock，不接真实后端 / 外部 API。

## 10. 明确不做什么
- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实模型推理。
- 不做生产级后端；不真实调用任何生成式 UI 模型（生成结果为预置 mock）。
- 不声称 grounding 判定是真实算法（Demo 中为预先标注，UI 明示）。

## 11. 成功体验标准
- 用户在 3 分钟内能说出：**它解决什么**（让生成的界面可信）、**怎么用**（裸/Pane 对照 + 点元素看溯源 + 动作预检）、**增量在哪**（Monogram 生成得更好；Pane 让生成得可信，是正交的一层）。
- 「裸 vs Pane」对照能制造清晰的顿悟：亲眼看到一个编造的价格 / 一个占位按钮，在裸模式下毫无破绽、在 Pane 模式下被拆穿。
- 对照组（食谱场景全 verified）证明 Pane 不是无脑报警器，而是「可信则背书、可疑则拦截」的可信层。
