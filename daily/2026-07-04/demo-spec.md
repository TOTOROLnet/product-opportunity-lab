# Demo 设计说明 — 2026-07-04

## 1. Demo 名称

**Attestor** — Agent 自述报告的证据审计器（反"幻觉式完成"验收台）

## 2. 一句话定位

把 AI Agent 交付时那段"我全做完了"的自信自述，逐句连到本次运行的真实证据（diff / 测试 / 命令 / 日志 / 截图），高亮"没有证据支撑"和"被证据反证"的完成声明，并把信任分从"它自己说的 100%"拉到"能被证明的真实值"。

## 3. 目标用户

审查 AI Agent 交付的人：用 Cursor / Copilot / Devin / Claude Code 把任务交给 agent、然后要 review/accept 其 PR 或任务结果的**开发者、代码 reviewer、QA / 平台工程师、对 Agent 交付质量负责的 AI PM / 团队 lead**。

## 4. 使用场景

Agent 报告"任务完成"，附带一段自然语言总结（"修复了 token 刷新竞态、补了边界用例测试、CI 全绿……"）。审查者面对两个坏选项：全信（可能吞下幻觉式完成）或全部亲手重验（抵消 agent 效率）。Attestor 给出第三条路：**逐句核验，只把"该亲自复核的少数几句"顶出来**。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / CLI / API / 基础设施类产品（验收/证据核验层，无可视 UI 的原生形态）
- 采用的演示方式：纯前端"**模拟体验 + 价值可视化**"——
  - **可点击的逐句证据审计**：自述里每条 claim 是一个可点击卡片，点开看它连到的证据（或"未找到证据"）与判定依据；
  - **before / after 信任分对比**：同一份交付，"自述信任分（它自己声称）100%"↔"证据信任分（被证明的）真实值"，直观展示增量价值；
  - **证据台（资源视图）**：把本次运行已有的 artifact 列出，显示哪些被 claim 引用、哪些是"孤儿证据 / 缺口"。
- 演示的是**我们分析出的创新切入点**（自述结论 ↔ 证据的逐条核验 + 幻觉式完成狙击），不是 Osloq（复现 bug 产出证据）或 Retrace（重放轨迹）的克隆。

## 6. 核心流程

（用户从进入到"看懂价值"的主路径，3 分钟内可完成）

1. **收件箱**：看到 3 个已完成的 Agent 交付（不同任务），每个显示 agent 的自信自述摘要 + 两个信任分徽章："自述 100%" vs "证据 XX%"，缺口越大越显眼。选一个进入。
2. **审计视图（主）**：左侧是 agent 的完整自述，每句 claim 被着色 + 带判定徽章（✅Verified / ◐Weak / ⚠Unsupported / ✖Contradicted）；点任意一句 → 右侧展开它连线到的证据（diff 片段 / 测试输出 / 命令退出码 / 日志 / 截图占位）与"为什么这么判"。顶部醒目告警条：**"N 条完成声明没有证据 / M 条被反证——重点复核这里"**。可切换 "原始自述（全绿假象）↔ 审计后（露出缺口）"。
3. **证据台**：列出本次运行的所有 artifact，标注每个被哪些 claim 引用；并列出"**声明缺口**"（有 claim 却无对应证据）与"**孤儿证据**"（有证据却没人声称），让用户理解信任分是怎么算出来的。

## 7. 页面结构（<= 3 个主要页面 / 视图）

- 视图 1：**Inbox（交付收件箱）** —— 3 个 mock 交付 + 自述/证据双信任分。
- 视图 2：**Audit（逐句证据审计）** —— 自述 claim ↔ 证据连线、判定徽章、告警条、before/after 切换。（核心）
- 视图 3：**Evidence（证据台）** —— artifact 清单 + 声明缺口 / 孤儿证据 + 信任分构成。

（单页应用内用顶部标签/路由在三视图间切换，不新开页面。）

## 8. 关键交互

- 点击自述中的任意 claim → 右侧证据面板联动高亮其证据与判定理由。
- 判定徽章筛选：只看 Unsupported / Contradicted（一键聚焦风险）。
- before/after 开关：隐藏判定 = 看到"agent 眼里的全绿"；打开 = 看到证据审计后的真实信任分（同一份交付，观感反差）。
- 信任分随筛选/交付变化实时重算并动画过渡。

## 9. 使用的 mock 数据

（全部为 mock，不接真实后端 / 外部 API / 模型）

- `runs`：3 个 Agent 交付，每个含：任务标题、agent 自述文本（拆成有序 claim 列表）、artifacts 列表（类型 ∈ diff/test/command/log/screenshot，含内容片段与元数据）、以及**预置的 claim→evidence 连线与判定**（Verified/Weak/Unsupported/Contradicted + 判定理由）。
- 判定为**确定性数据**（在 mock 中写死），前端仅做展示与聚合（把"验证引擎"抽象为可解释的规则说明，避免假装有真实 LLM 在跑）。
- 信任分算法：`证据信任分 = 加权(Verified=1, Weak=0.5, Unsupported=0, Contradicted=-* 惩罚)`，在代码中显式计算，规则在证据台可见。

## 10. 明确不做什么

- 不做登录 / 用户系统
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实 LLM 调用
- 不做生产级后端；不真正解析真实 PR 或运行真实测试
- 不复现 bug、不重放执行轨迹（那是 Osloq / Retrace 的事，本 Demo 只做"自述 ↔ 证据核验"）

## 11. 成功体验标准

- 用户 3 分钟内能说出："它解决什么（核验 agent 自述有没有证据）、怎么用（点交付→看逐句判定→聚焦无证据/被反证的声明）、增量在哪（自述 100% vs 证据 XX% 的反差，直接定位该复核哪几句）"。
- 首屏即传达"自称完成 ≠ 被证明完成"的核心张力；至少一条 claim 是 Contradicted（被反证）以制造"幸好查了"的顿悟感。
