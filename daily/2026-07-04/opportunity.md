## 1. 报告事实 vs 我的独立判断

| 维度 | 报告怎么说（参考资料） | 我作为创业者的独立判断 |
| --- | --- | --- |
| 核心瓶颈在哪 | 瓶颈在"**复现**"——issue 是否可复现、失败路径在哪，Osloq 把复现产品化 | 部分认同，但我认为复现只是证据链的**上游**。更普遍、更高频的痛点在**下游的"验收"**：Agent 干完活会给一段**自信满满的自述**（"已修复 / 已加测试 / CI 全绿"），而**审查者无法快速分辨哪句有证据、哪句是乐观脑补**。复现解决"能不能重现问题"，却没解决"**Agent 说完成了，到底成没成**"。 |
| 证据是谁的证据 | Osloq 产出的是**它自己复现过程**的 evidence timeline | 我质疑这个隐含前提。真正稀缺的不是"再产出一份证据"，而是把**Agent 已有的产出（自述结论）与已有的证据（diff / 测试输出 / 命令退出码 / 日志 / 截图）对齐**，指出**哪些结论没有任何证据背书**。证据的价值在于**核验别人的主张**，而不只是自证。 |
| verdict 的对象 | verdict = "这个 bug 可否复现" | 我认为下一个更值钱的 verdict 是 "**这个 Agent 的完成声明可否被证据支撑**"。coding agent、browser agent、本地操作 agent 都会给结论，但**没有产品把"自然语言结论 ↔ 具体证据"逐条连线并给出可信度**。 |
| action manifest / 可审查性 | Glaze 提出应有"人工可审查的 action manifest" | 认同方向，但 manifest 只是"它**打算/被授权**做什么"（意图/权限）。我关注的是更靠后的一层："它**声称做完了什么** vs 证据显示它**实际**做了什么"——即**声明与证据的差距（claim-evidence gap）**，这才是接管决策的直接依据。 |
| Context Engineering / 控制面 | 重点在"证据层/记忆层/审批层" | 报告反复出现"证据 / verdict / replay / 需要人接管"，却**没有产品把"Agent 的自述报告"当作一等对象来做证据核验**。这正是 `lab-focus.md` 明列的"高风险工具调用的控制面 / AI 原生生产力 / Coding Agent"方向上一个**被信号环绕却无人正面做**的空位。 |

结论：报告的叙事集中在"**产出**证据（复现）"。真正被忽视、又高频命中痛点的**单点楔子**是——**把 Agent 交付时的自述结论，逐条拆成可核验的"声明（claim）"，与本次运行已有的证据 artifact 自动连线，给每条声明判定"有证据支撑 / 证据薄弱 / 无证据 / 与证据矛盾"，把一段"看起来 100% 完成"的自述，变成一张"有多少完成是真的被证明了"的信任地图**。

## 2. 候选机会（至少 3 个，均为创新切入点，非照抄报告中的产品）

### 候选 A：Attestor — Agent 自述报告的"证据审计器"（反"幻觉式完成"验收台）
- 创新切入点：不复现、不重放、不审批。取 Agent 交付时的**最终自述**（PR 说明 / 任务总结 / "我做了什么"），用**声明抽取**把它拆成一条条可核验的 claim（如"修复了 token 刷新竞态"、"新增了边界用例测试"、"CI 全绿"），再把每条 claim **连线到本次运行已有的证据**（代码 diff / 测试输出 / 命令退出码 / 日志 / 截图），逐条判定 **Verified（有证据）/ Weak（证据薄弱）/ Unsupported（找不到证据）/ Contradicted（证据反证）**，并把 **Unsupported / Contradicted 顶到最显眼处**——那正是"幻觉式完成"藏身之地。给出 before/after：**自述信任分 100%（它自己说的）→ 证据信任分 XX%（能被证明的）**。
- 目标用户 / 场景：审查 Cursor / Copilot / Devin 等 coding agent 交付（PR / 任务结果）的**开发者、reviewer、平台/QA 工程师、对 Agent 交付负责的 AI PM**——在**接受（accept）之前**快速定位"哪几句需要亲自复核"，而不是要么全信、要么全部重验。
- 五维评分：痛点 5/5，机制新意 4/5，方向相关 5/5，一天可行性 5/5，启发价值 5/5 → 合计 **24/25**
  - （诚实校准：机制新意给 4——"引用核验 / 事实核查 / 溯源"范式在其他领域已有先例；**把它迁移到"Agent 自述结论 ↔ 运行 artifact"、并专门狙击"幻觉式完成声明"是新的应用切入点**，而非全新算法。）

### 候选 B：Ledger — 本地操作 Agent 的"飞行记录仪"（副作用证据 + 可逆性清单）
- 创新切入点：受 Glaze"本地权限 / action manifest"启发，但换到**执行之后**：记录 Agent（或 Agent 生成的本地应用）**实际**对真实机器做了什么——文件读写 before/after diff、网络外呼、进程/命令、注册表级改动——按**风险 / 可逆性**分级，产出一份"到底改了什么"的**可审查回执**与"需要人接管"建议。
- 目标用户 / 场景：跑本地操作 / Computer Use / 桌面 Agent 应用的团队。
- 五维评分：痛点 4/5，机制新意 4/5，方向相关 5/5，一天可行性 4/5，启发价值 4/5 → 合计 **21/25**
  - （与我此前 Fusebox 的"副作用 / 可逆性"及 Retrace 的"重放"有部分气质重叠，且本地 OS agent 当下受众偏窄；纯前端 mock OS 级副作用说服力略弱于 A。）

### 候选 C：Dedup — 复现失败的"根因签名聚类"（issue triage 去重）
- 创新切入点：Osloq 逐个复现单 issue，但真实 triage 是**issue 洪流**，大量近似重复。Dedup 对已复现的失败按**根因签名**聚类，让团队"**改一次修一片**"，而非逐条 triage 30 份近似报告。
- 目标用户 / 场景：大型 OSS / 支持团队的 issue triage。
- 五维评分：痛点 4/5，机制新意 3/5，方向相关 4/5，一天可行性 4/5，启发价值 3/5 → 合计 **18/25**
  - （按签名聚类在错误监控（Sentry 式 error grouping）里已很成熟，迁到 Agent 复现有价值但机制新意有限，且强依赖上游"已能稳定复现"。）

## 3. 最终选择

- 选中：**候选 A — Attestor（Agent 自述报告的证据审计器 / 反幻觉式完成验收台）**，总分 **24/25**
- 为什么选它（相对 B、C 的优势）：
  1. **痛点最真实、最当下、最高频**：只要你用过 coding agent，就一定遇到过"它自信地说'全部完成、测试通过'，结果一查根本没加测试 / CI 其实红了"。**信任那段自述很危险，逐条重验又抵消了 Agent 的效率**——这正是每天都在发生、且随 Agent 交付量上升越来越痛的矛盾。报告作者本人也把"用户还要知道哪些证据支持结论"列为最想跟进方向。
  2. **最适合轻量 Demo**：核心体验 = "看自述 → 逐句连线证据 → 高亮无证据/被反证的声明 → 信任分 before/after"，天生是**可视化 + before/after + 可点击核查**题材，纯前端 + mock 一次运行（自述 + artifact + 预置连线）即可把价值讲透（属 loop 的"抽象/基础设施类 → 模拟体验 + 价值可视化"型）。
  3. **与 lab 关注方向命中最准、且与本仓库旧产物分属不同层**：命中"Coding Agent / AI 原生生产力 / 高风险工具调用的控制面"；且它是**执行之后的"验收/核验层"**——与 Gavel（动作发生时审批）、Contextlens（决策形成前的上下文）、Fusebox（授权时的能力面组合）、Osloq/Retrace（复现/重放执行轨迹）都不同层，互补而非重叠。
  4. 相比 B（受众偏窄、与旧产物气质重叠）、C（机制新意与受众都弱、强依赖上游复现），A 的独立性、当下性与可演示性都更强。

## 4. 深入论证（针对 Attestor）

1. 目标用户：**审查 AI Agent 交付的人**——用 Cursor / Copilot / Devin / Claude Code 等把任务交给 agent、然后要 review/accept 其 PR 或任务结果的开发者、代码 reviewer、QA / 平台工程师，以及对"Agent 交付质量"负责的 AI PM / 团队 lead。
2. 核心痛点（真实、高频、可付费）：Agent 交付的**结论是自然语言、且天然偏乐观**（被训练成"给出令人满意的总结"）。审查者面对一段"已修复/已加测试/已验证"的自信自述，**只有两个坏选项：全信（可能吞下幻觉式完成）或全部亲手重验（抵消 agent 的效率）**。缺一个东西：**把每句结论和它该有的证据逐条对齐，直接告诉我"这几句没有证据 / 这句被反证，重点复核这里"**。这类"验收/核查"是团队愿意付费的（CI、code review、QA 工具都在这条价值线上）。
3. 现有产品为什么没解决好：**Osloq** 解决"能不能**复现**问题"（上游、且只针对 bug issue）；**Retrace** 解决"**重放**已发生的失败轨迹、哪步错了"（调试）；**Gavel / Basedash** 解决"动作发生**时**要不要**批准**"（运行时闸门）；**Glaze 的 action manifest** 描述"被授权/打算做什么"（意图/权限）。**没有一个把"Agent 交付时的自述结论"当作一等对象，逐条核验它与真实证据的差距，并狙击"无证据的完成声明"。**
4. 为什么适合做轻量 Demo：核心体验是"**读一段 agent 自述 → 系统把每条 claim 连到 diff/测试/命令/日志/截图 → 高亮 Unsupported/Contradicted → 看信任分从 100% 掉到真实值**"，三视图（收件箱 / 审计视图 / 证据台）、纯前端、全 mock（自述 + artifact + 确定性连线规则）即可闭环，无需真实后端、真实凭证或模型。

## 5. 不照抄声明（四项必填）

1. **创新切入点是什么**：把 **Agent 交付时的自述结论**拆成可核验的 **claim**，与本次运行**已有**的证据 artifact（diff / 测试 / 命令退出码 / 日志 / 截图）**逐条连线**，给出 **Verified / Weak / Unsupported / Contradicted** 判定，专门把"**没有证据背书的完成声明**（幻觉式完成）"顶出来，并用 **before/after 信任分**量化"自称完成"与"被证明完成"的差距。
2. **解决的具体问题 / 场景**："Agent 说全都做完了、还很自信，我该全信、还是全部重验？"——Attestor 直接指出"这几句有证据、这几句没证据、这句被反证，重点复核这里"，把验收从"凭感觉信任"变成"看证据信任"。
3. **相对被参考产品的增量价值（差异点）**：
   - vs **Osloq**：Osloq **产出**证据——在 sandbox 里**复现** bug、输出复现 verdict + evidence timeline（上游、只针对 bug issue、证据是它自己复现过程的证据）。Attestor **不复现、不跑任何东西**，而是**消费** Agent 已有的自述 + 已有的 artifact，**核验每条结论是否被证据支撑**（下游验收、适用于任意 agent 的任意交付、证据是用来核验"别人的主张"）。Osloq 答"这个 bug 能不能复现"，Attestor 答"这个 agent 说的'完成'成不成立"。
   - vs **Retrace**：Retrace **重放**失败轨迹找"哪步错了"（调试执行过程）；Attestor 核验"**声称的成功到底是不是真的**"（在一次可能成功的运行之上做声明↔证据对齐），不重放步骤。
   - vs **Gavel / Basedash（动作审批）**：它们在动作**发生前/时**做 approve/deny（运行时闸门）；Attestor 在 agent **报告完成之后**做事后审计（核验叙事），不是运行时闸门。
   - vs **Fusebox（本仓库旧产物）**：Fusebox 在**授权/部署前**分析"权限组合能构成哪些高危链路"；Attestor 在**执行之后**分析"自述 vs 证据的差距"，完全不同层。
   - vs **Glaze 的 action manifest**：manifest 描述"**打算/被授权**做什么"（意图）；Attestor 核验"**声称做完了什么** vs 证据显示的**实际**结果"（声明与证据的差距）。
4. **为什么这不是照抄 X（本质区别）**：报告里**没有任何产品**把"Agent 的**自述结论**"当作一等对象、去做**逐条 claim↔证据连线 + 无证据/反证声明狙击 + before/after 信任分量化**。报告中的产品要么在产出复现证据（Osloq），要么在重放轨迹（Retrace），要么在运行时审批（Gavel/Basedash），要么在声明意图（Glaze manifest）。Attestor 的核心创新（自述结论的证据审计 + 幻觉式完成检测 + 信任分 before/after）在报告任一产品中都不存在，因此不是克隆；它正是报告作者所说"哪些证据支持结论、哪里需要人接管"的一个**可落地原语**。

## 6. 不确定性与风险

- 需求侧风险：如果多数团队本来就要求"agent 必须自带绿灯 CI 才能合"，那"声明核验"可能被视为"CI 已经做了"。对冲：CI 只覆盖"可自动化验证的一小部分声明"（能编译、测试通过），而 agent 自述里大量是**CI 覆盖不到的定性声明**（"我处理了边界情况"、"我没改公共 API"、"我加了日志"）——这些恰恰是幻觉高发区，也是 Attestor 的主战场。
- 落地 / 技术风险：真实产品要把"自然语言 claim"稳定拆解、并可靠地连到正确证据，本身就需要不弱的抽取/对齐能力；"证据支撑度"判得太松会漏报、太严会满屏告警造成新的疲劳。判定标准若不透明会失去信任。Demo 用**确定性的连线规则 + 显式判定依据**规避，但这是真实落地最大的准确率/信任挑战。
- 我可能判断错的地方：也许审查者更想要的是"**直接帮我把没证据的部分补上证据（自动跑测试）**"，而不是"**告诉我哪里没证据**"。若如此，Attestor 需要从"审计器"进一步长成"审计 + 取证"闭环（与 Osloq 式复现能力互补）。这说明 A 与 Osloq 长期是**互补**关系，而非竞争——更印证它不是照抄。

## 7. 门槛结论

- 最高分：**24/25**
- 是否达到 16/25 门槛：**是**
- 结论：**进入 Demo 开发**（属"抽象/基础设施类"，采用"模拟体验 + 价值可视化"的可点击交互 Demo：收件箱 → 逐句证据审计 → 证据台 + 信任分 before/after）。
