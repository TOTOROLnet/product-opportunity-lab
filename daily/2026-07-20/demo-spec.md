# Demo 设计说明 — 2026-07-20

## 1. Demo 名称

知时 Aptly — 该出声时才出声的环境记忆层

## 2. 一句话定位

垫在任意「环境记忆源」（如 Rewisp）之上的**主动召回决策层**：用相关度×时效×可行动性打分 + 打扰预算闸门 + 去重 + 过时抑制，决定此刻是把某条旧记忆推到你眼前、还是**保持安静**——把"提醒一切"变成"该提醒时才提醒"。

## 3. 目标用户

日均在 20+ App / 标签页间高频切换、"看过就忘、忘了还不自知"的知识工作者与独立开发者（先服务个人/prosumer，后以 MCP 接入 agent）。

## 4. 使用场景

一个普通上午：给 Acme 回信时忘了 20 分钟前看到"deadline 提前到本周四"；改退款代码时忘了"退款 API 参数刚改名"、又差点用了已被取代的旧退款上限；答应周五发报价却漏掉。知时在这些**关键时刻**只推该推的那一两条，其余保持安静。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / CLI / API / 基础设施类产品（一个"决策层/判断引擎"，不是可视化终端产品）
- 采用的演示方式：纯前端「**模拟体验 + 价值可视化**」——
  - **脚本化时间线回放**：把"一个上午"的记忆流 + 活动流脚本化，逐步回放；
  - **before / after 对比**：同一条 trace 分别用「朴素推送」与「知时」两种策略跑，指标由确定性引擎实测；
  - **玻璃盒**：把每个"现在"对每条候选记忆的打分与决策全部摊开。
- 演示的是**我们分析出的创新切入点**（有节制的 PUSH 决策层），不是 Rewisp（记忆存储 + PULL 查询）的克隆。

## 6. 核心流程（3 分钟看懂价值）

1. 进入「① 工作台」，默认停在 09:15「改退款代码」——最能说明问题的关键时刻。看到知时主动提示了 2 条（退款 API 参数改名 + 退款上限变更），并**替你拦下**了 1 条过时的旧 SOP。
2. 点「朴素推送」切换策略：同一时刻它把**过时的旧 SOP（¥500）**也推了出来（红色"已过时"），且总打扰更多、精确率更低。
3. 用「播放 / 下一步」走完整个上午，观察累计指标（打扰 / 有用 / 精确率 / 过时误导 / 关键覆盖）。
4. 进「② 决策透视」，看知时"为什么这条推、那条不推"（相关度、综合分、决策、原因）。
5. 进「③ 对比」，一眼看到两种策略的实测差距。

## 7. 页面结构（<= 3 个主要页面）

- 页面 1 · **工作台（时间线回放）**：模式切换 + 播放/单步 + 记忆库 + 时间线 + 此刻的提示卡 & 累计指标。
- 页面 2 · **决策透视（玻璃盒）**：每个活动对每条候选记忆的相关度/综合分/决策/原因表；可按记忆筛选。
- 页面 3 · **对比**：朴素 vs 知时 并排，6 项实测指标 + 每步行为差异 + 结论解读。

## 8. 关键交互

- 模式切换（朴素推送 / 知时）实时改变提示与指标。
- 时间线播放 / 上一步 / 下一步 / 重置。
- 记忆库随时间"逐渐出现"，被提示的记忆打上"已提示"徽章，过时的打上"过时"。
- 玻璃盒按记忆 id 点选筛选。

## 9. 使用的 mock 数据

- `src/data/scenario.ts`：7 条环境记忆（含 1 条被取代的过时项 M7、1 条低价值噪音 M3/M5、3 条高价值变更/承诺）、7 个活动（now）、主题关联表、策略参数。
- 全部为 mock 与脚本化时间线，**不接**真实后端 / LLM / agent / 外部 API。相关度用确定性代理函数模拟。

## 10. 明确不做什么

- 不做登录 / 用户系统
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实 LLM / 真实屏幕抓取
- 不做生产级后端
- 不声称真实召回质量——只演示"机制成立 + 价值形态"

## 11. 成功体验标准

- 用户 3 分钟内能说出："它在正确的时刻只推该推的，替我拦下噪音与过时信息；比无脑提醒更少打扰却更不漏关键。"
- 指标可复现：`src/logic/engine.ts` 为确定性引擎，同一 trace 每次跑出的数字一致（朴素 8 打扰/38%/3-4；知时 4 打扰/100%/4-4）。
