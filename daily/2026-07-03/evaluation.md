# 自动体验 / 自测报告 — 2026-07-03

> 以 AI 产品经理 + 目标用户（Agent 开发者 / Context Engineer）双重视角，对 Contextlens Demo 做一次体验检查。结论只能是 PASS / PARTIAL / FAIL。

## 1. 构建结果
- `npm install`：**成功**（67 packages）。
- `npm run build`：**成功**（`tsc && vite build`，第 1 轮通过，无类型错误；失败修复轮次 0/3）。
- smoke 检查：**通过**（`dist/index.html` 非空、含 `id="root"` 挂载节点、生成 1 个 JS bundle；`validate_demo.sh` 返回 OK）。

## 2. 可用性检查
- `npm run preview` 本地 4173 端口返回 HTTP 200，资源以相对路径 `./assets/...` 加载（`base:'./'` 生效，可部署到 Pages 子路径）。
- 用浏览器实际打开验证：**首屏非空**，深色 UI、头部、三 Tab、场景切换、时间线、上下文窗口面板均正常渲染；控制台仅有一个无害的 `favicon.ico` 404，无 JS 运行时错误。

## 3. 核心流程是否闭合
**闭合。** 已实机走通：
1. Run X 光（基线，默认停在第 4 步）→ 看到关键『退款政策』被驱逐（置灰 + 删除线 + 原因 + "已驱逐"标签）。
2. 点第 6 步 → 右侧更新为危险结论：直接放款 $900、未触发审批。
3. 修复对比 → 左右并排：基线 99% 用量 / 事故；修复 63% 用量、策略 Pin 在窗口 / 安全转人工审批。
4. X 光报告 → 三张成绩单卡片（驱逐 2 / 缺失关键约束 2 / 可压缩冗余 9,800）+ 修复建议。
5. 场景切到"修复" → 指标全部归 0、结论转绿。

## 4. 首屏是否讲清产品价值
基本讲清。首屏顶部红色"结论"条 + 第 4 步的驱逐画面能快速传达"关键约束被静默挤出窗口 → 导致危险动作"。若要更快切入，可考虑默认落在"修复对比"页做首屏，但当前 3 分钟脚本已能覆盖。

## 5. 交互是否有断点
- 无功能性断点：Tab 切换、时间线选步联动、场景切换均即时生效。
- 小瑕疵：缺 favicon（404，不影响功能）；时间线较窄时长标题会省略（已用 ellipsis 处理，属预期）。

## 6. 是否只是概念包装（自嗨检测）
- 是否引用报告信号：**是**。明确引用报告事实（Retrace 的 fork/replay + prove-the-fix、scritty 的可搜索记忆、Basedash 的审批 payload），并在 opportunity.md 用表格区分"报告事实 vs 我的独立判断"。
- 是否明确创新切入点与增量价值：**是**。切入点=把"上下文窗口本身"变成可观测/可对比/可修复对象，并做"驱逐/淹没 → 行为回归"的因果关联；已列出 vs Retrace / scritty / Basedash(Gavel) 的差异。
- 是否是某产品的照抄：**否**。报告中无任何产品把上下文窗口作为一等可观测对象；本 Demo 演示的是我们独立分析出的切入点，且刻意区别于本仓库旧示例 Gavel（审批驾驶舱，动作层）——Contextlens 在认知层。

## 7. Demo 的最大问题
数据是精心设计的 mock，"驱逐/压缩"的因果被刻意做得干净漂亮；真实产品最难的一步是**从各 Agent 框架采集"每步实际送入模型的完整上下文与驱逐/截断细节"**，各框架差异大、很多不暴露该信息。Demo 用 mock 规避了这一工程挑战——它证明的是"价值主张成立且可视化清晰"，而非"采集可行"。

## 8. 是否建议用户人工体验
**建议**。核心价值（before/after 的"事故→合规"对比）在真实点击中冲击力最强；本次已用浏览器实测通过并留存 3 张截图（`screenshots/`）。

## 9. 下一步建议
1. 做一个"真实采集"的技术探针：从 1 个开源 Agent 框架（如 LangGraph / OpenAI Agents SDK）导出每步上下文，验证"每步窗口重建"是否可行。
2. 增加"淹没/lost-in-the-middle"维度（不止驱逐），覆盖大窗口时代的相关性问题。
3. 与审批类产品（如本仓库旧示例 Gavel）做互补集成：认知层（为什么这么决定）+ 动作层（要不要放行）。
