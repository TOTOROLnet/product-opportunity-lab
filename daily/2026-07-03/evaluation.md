2. **风险链路**：列出 5 条候选链路，按残余风险排序（2 高危 / 2 中危 / 1 低危）；展开首条显示「评分构成」逐项拆解 + 绿色「建议修复」框。

## 6. 是否只是概念包装（自嗨检测）

- 是否引用了报告中的具体信号：**是**。切入点直接建立在报告的 Basedash Actions（Always allow / Needs approval / Blocked 三态、逐动作 payload 审批）、PieterPost/Macuse（MCP 高风险动作）、Retrace（事后重放）等具体信号之上，并作了区分与反驳。
- 是否明确了创新切入点与增量价值：**是**。切入点=把"权限集合"当一等对象做"组合涌现风险"分析；增量=爆炸半径/可逆性量化 + 部署前 before/after 模拟，明确区别于逐动作运行时审批与事后重放。
- 是否是某产品的照抄：**否**。报告中没有任何产品把"授予的工具权限集合"当作一等对象做组合链路风险发现；与本仓库旧示例 Gavel（动作层审批）、Contextlens（上下文认知层）分属不同层。

## 7. Demo 的最大问题

- **评分是显式规则的 mock，非真实度量**：爆炸半径/可逆性权重是我拍的演示值，真实产品需要基于工具语义与真实数据流校准，否则易被质疑"分数怎么来的"。Demo 已把评分构成透明展示以缓解，但仍是最大 gap。
- **总风险初始被封顶到 100**：初始多条高危叠加后归一化触顶，损失了一点区分度（100 更像"爆表"而非精确值）；before/after 的相对变化不受影响，但绝对值的头部区分度偏弱。
- **真实落地的取数难题未在 Demo 体现**：真实产品要拿到"某 Agent 实际被授予的完整工具/MCP 清单 + 读/写/外发语义 + 数据敏感度标注"，跨框架标准化难——这是 opportunity.md 已指出、但 Demo 用 mock 规避的核心工程风险。

## 8. 是否建议用户人工体验

建议。核心价值（"改一处权限 → 看全局高危链路收敛"）在真实点击中体感最强；静态截图只能看到状态，看不到 before/after 的实时收敛动效。

## 9. 下一步建议

1. 引入"事故案例库"：把每类高危链路配一个真实世界的失败故事，增强"运行时逐动作审批来得太晚"的说服力。
2. 评分可解释性升级：让权重可调 + 给出敏感度分析，把"拍的分"变成"可辩护的分"。
3. 探索真实取数路径：先做一个从 MCP server 列表 / Agent 配置文件导入工具清单的 importer（仍纯读、不执行），验证"能力面"是否可自动重建。
4. 归一化调参：让初始基线落在 ~90 而非触顶 100，恢复头部区分度。
