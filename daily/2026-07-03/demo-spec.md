# Demo 设计说明 — 2026-07-03

## 1. Demo 名称

**Fusebox** — Agent 能力"配电箱"：高危工具链路的部署前控制面。

## 2. 一句话定位

把 Agent 授予的一堆 MCP/工具权限画成一张"能力图"，自动标出"单看安全、组合起来高危"的涌现链路（数据外泄 / 破坏性写入 / 不可逆动作），并让你在部署前模拟收紧权限、看高危链路怎么收敛。

## 3. 目标用户

给生产 Agent 接入多个 MCP/工具（数据库、支付、邮件、Slack、内部文档、Computer Use 等）的开发者 / 平台工程师 / 安全工程师 / 对 Agent 行为负责的 AI PM。

## 4. 使用场景

我给一个"客服自动化 Agent"接了 7 个工具：读客户库、跑只读 SQL、跑写 SQL、发邮件(Resend)、发起 Stripe 退款、发 Slack、读内部文档。每个工具我都按直觉设了 Always allow / Needs approval / Blocked。上线前，我想知道：**这组权限被 Agent 串起来，能构成哪些我没预料到的高危链路？我该收紧哪一个，才能用最小代价拆掉最大风险？**

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / CLI / API / 基础设施类产品（高风险工具调用的**控制面 / 权限层**）
- 采用的演示方式：纯前端"**模拟体验 + 价值可视化**"——
  - **资源/结构视图**：把抽象的"工具权限集合"可视化成一张能力图（数据源 → 工具 → 外部副作用）。
  - **风险路径列表**：把系统发现的涌现高危链路排序展示，每条附爆炸半径 / 可逆性 / 数据敏感度评分与人类可读解释。
  - **before / after 对比**：策略模拟器里改一条权限/加一条防火墙规则，实时对比"高危链路数、总风险分"的前后变化。
- 演示的是**我们分析出的创新切入点**（权限"组合"涌现风险 + 部署前收敛），不是被参考产品（Basedash 逐动作审批 / Retrace 事后重放）的克隆。

## 6. 核心流程（3 分钟内可完成）

1. **进入即看懂**：首屏是一张客服 Agent 的能力图 + 顶部风险横幅（"发现 4 条高危链路，总风险分 78/100"）。左图右栏（Inspector）。
2. **看图**：节点分三层——数据源（客户库、内部文档）/ 工具（只读 SQL、写 SQL、发邮件、退款、Slack）/ 外部副作用（客户收件箱、公网、账务）。系统自动标红的**高危复合链路**在图上高亮成一条会流动的路径。
3. **看路径**：切到"风险链路"标签，看排序后的链路，例如：
   - `读客户库 → 发邮件` = **数据外泄**（敏感 PII 可离开组织，不可逆）
   - `跑写 SQL（无 WHERE 护栏）` = **破坏性写入**（可逆性差，除非有备份）
   - `Stripe 退款（无金额上限）` = **财务风险**（部分可逆、爆炸半径大）
   - 点任意链路 → Inspector 显示评分构成、涉及的工具、为何危险、建议修复。
4. **模拟收敛**：切到"策略模拟器"，把"发邮件"从 Always allow 改成 Needs approval，或给"客户库→公网"加一条数据防火墙规则 → 顶部风险分与高危链路数**实时下降**，并给出 before/after 对比条。
5. **得出结论**：用户看懂"这组权限的爆炸半径 / 最该收紧哪一个 / 收紧后风险怎么降"。

## 7. 页面结构（<= 3 个主要视图，单页应用内 Tab 切换）

- 视图 1：**能力图（Capability Graph）** — SVG 能力拓扑 + 高危链路高亮 + 节点/边 Inspector。
- 视图 2：**风险链路（Risk Paths）** — 涌现高危链路排序列表 + 每条的评分与修复建议。
- 视图 3：**策略模拟器（Policy Simulator）** — 工具三态开关 + 数据防火墙规则开关 + before/after 风险对比。

## 8. 关键交互

- 点击能力图节点/边 → 右侧 Inspector 展示语义（读/写/外发、数据敏感度）。
- 点击某条高危链路 → 图上对应路径高亮 + Inspector 展示评分拆解与修复建议。
- 策略模拟器里切换任一工具策略 / 防火墙规则 → 全局风险分、高危链路数、图上高亮**实时重算**（纯前端确定性计算）。
- 顶部"风险横幅"随策略变化实时更新，并显示相对初始基线的 before/after。

## 9. 使用的 mock 数据

全部为 mock，不接真实后端 / 外部 API / 凭证：

- `scenario.ts`：一个客服 Agent 的能力拓扑——
  - **节点**：数据源（`customer_db` 高敏、`internal_docs` 中敏）、工具（`sql_read`、`sql_write`、`send_email`、`stripe_refund`、`slack_post`）、外部副作用汇（`customer_inbox`/公网、`ledger`/账务、`internal_channel`）。
  - **每个工具**：类型（读/写/外发）、初始策略（Always allow / Needs approval / Blocked）、是否可逆、护栏缺失标记（如写 SQL 无 WHERE 护栏、退款无金额上限）。
  - **数据敏感度**：PII / 财务 / 内部 / 公开。
- **风险规则（显式、可解释，非黑箱）**：如"高敏数据源可达外发工具 = 数据外泄"、"不可逆写工具且无护栏 = 破坏性"、"财务外部副作用无上限 = 财务风险"。每条链路的分数 = 爆炸半径 × 可逆性 × 数据敏感度的加权（规则写在 `scoring.ts`，展示给用户看）。

## 10. 明确不做什么

- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 邮件 / 外部私钥 / 真实 MCP server / 真实 API。
- 不做生产级后端、不做真实模型调用。
- 不声称已验证真实产品的能力；所有拓扑与评分均为演示用 mock + 显式规则。

## 11. 成功体验标准

- 用户 3 分钟内能说出："它解决什么问题（部署前发现工具权限的组合高危链路）、怎么用（看图 → 看链路 → 模拟收紧）、增量在哪（组合涌现风险 + 爆炸半径/可逆性量化 + 部署前 before/after，而非逐动作运行时审批）"。
- 策略模拟器能让"高危链路数 / 总风险分"发生**可见且可解释**的前后变化，让用户直观体会"收紧哪一个最划算"。
