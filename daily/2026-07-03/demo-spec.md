# Demo 设计说明 — 2026-07-03

## 1. Demo 名称
Gavel — Agent 动作审批驾驶舱

## 2. 一句话定位
让高风险 Agent 动作在几秒内被安全裁决——附带决策上下文，且每次裁决都会沉淀为策略。

## 3. 目标用户
为"会执行动作"的 Agent 后果负责的人：运营、数据、财务、增长、PM。

## 4. 使用场景
Agent 提出一批高风险动作（退款、批量删库、群发营销、跨系统同步…），运营者需要快速、安全地逐条裁决，并让系统从裁决中学习。

## 5. Demo 策略分型
- 本机会属于：☑ 可视化/交互类产品（审批面天生是 UI 产品）
- 采用的演示方式：直接做**可点击交互 Demo**（真实可操作：批准/改后批准/拒绝会改变收件箱与策略状态）。
- 演示的是我们的创新切入点（决策上下文装配 + 裁决即策略），不是 Basedash 等的克隆。

## 6. 核心流程（3 分钟内看懂价值）
1. 进入 **审批收件箱**：看到待裁决的高风险动作队列，每条带风险等级、发起 Agent、影响范围摘要。
2. 点开一条 → **决策详情**：看到"决策上下文卡片"——Agent 推理链摘要、精确 payload、影响范围/可回滚性、历史相似裁决（含一条被拒的理由）。
3. 做出裁决：**批准 / 改后批准 / 拒绝**。拒绝时可"一键转为策略"。
4. 打开 **策略与学习**：看到由历史裁决自动沉淀的策略，以及"本周被策略自动处理的动作数 / 平均裁决耗时"等指标——直观体现"裁决即策略、越用越省"。

## 7. 页面结构（<= 3 个主要页面）
- 页面 1：审批收件箱（Inbox）
- 页面 2：决策详情（Decision Detail，抽屉/详情面板）
- 页面 3：策略与学习（Policies & Learning）

## 8. 关键交互
- 收件箱条目点击 → 详情面板。
- 详情内三个裁决按钮改变该条状态并从待办移除；拒绝可生成策略。
- 策略页展示自动/手动策略；顶部指标随裁决动态更新（mock 计算）。
- 顶部导航在三页间切换。

## 9. 使用的 mock 数据
- `pendingActions`：若干高风险动作（含 agent、风险级别、payload、推理链步骤、影响范围、相似历史裁决）。
- `policies`：预置几条策略（自动/人工）。
- 全部为前端 mock，不接任何后端 / 数据库 / 外部 API。

## 10. 明确不做什么
- 不做登录 / 用户系统
- 不接数据库 / 支付 / 外部私钥 / 真实 Agent / 真实 API
- 不做生产级后端
- 不做真实的动作拦截与推理链采集（真实落地的难点，Demo 用 mock 表达）

## 11. 成功体验标准
- 用户 3 分钟内能说出：Gavel 解决"高风险 Agent 动作的橡皮图章 vs 瓶颈"，用"决策上下文 + 一键裁决"，增量在"裁决即策略、越用越省"。
- 能真实点击完成"看队列 → 读上下文 → 裁决 → 看策略沉淀"的闭环。
