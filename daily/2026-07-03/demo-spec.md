# Demo 设计说明 — 2026-07-03

## 1. Demo 名称
Contextlens — Agent 上下文窗口"X 光片"

## 2. 一句话定位
让 Agent 的"脑内上下文"变得可见：逐步透视上下文窗口里装了什么、什么被截断/驱逐/淹没，以及它如何直接导致了错误或危险行为——再一键 Pin/压缩看修复对比。

## 3. 目标用户
构建多步 / 长任务 / 带工具与记忆的 Agent 的开发者、Context Engineer、对 Agent 行为负责的 AI PM。

## 4. 使用场景
一个客服/账务 Agent 在处理"退款 $900"请求的 8 步执行中，突然违反了"单笔 > $500 需人工审批"的策略、直接放款。用户打开 Contextlens 复盘：发现该策略块在第 4 步因窗口预算被驱逐，导致第 6 步失去约束 → 定位真因并验证修复。

## 5. Demo 策略分型（重要）

- 本机会属于：☑ 抽象 / 基础设施类产品（上下文/Context Engineering 可观测性）　□ 可视化/交互类产品
- 采用的演示方式：纯前端"**模拟体验 + 价值可视化**"。
  - **模拟轨迹回放**：脚本化重放一条 Agent 执行轨迹（逐步）。
  - **窗口结构视图**：把抽象的"上下文窗口构成 + 驱逐/淹没"可视化为带 token 占比的堆叠块。
  - **before / after 对比**：切换"基线 vs 修复（Pin 策略 + 压缩工具输出）"，直观看到"危险动作 → 被安全拦下"的增量价值。
- 演示的是**我们分析出的创新切入点**（窗口 X 光 + 驱逐因果 + 修复对比），不是 Retrace/scritty/Basedash 的克隆，也不同于本仓库旧示例 Gavel（审批驾驶舱）。

## 6. 核心流程（3 分钟内看懂价值）
1. 首屏读到定位与"基线/修复"切换；看到一条 8 步执行时间线，第 6 步带红色高风险标记。
2. 点击时间线上的步骤 → 右侧"上下文窗口 X 光"展示该步窗口构成：各块按 token 占比堆叠，被驱逐/截断的块置灰并标注原因（预算溢出 / 相关性过低 / 被工具输出挤占）。
3. 点到第 4 步 → 看到"退款政策"块被驱逐的事件卡；点到第 6 步 → 看到 Agent 因"策略不在窗口"而放款的因果连线。
4. 切换到"修复"场景（Pin 策略 + 压缩工具输出）→ 同一第 6 步，策略仍在窗口，Agent 改为"触发需审批、暂停" → 危险动作被拦下。
5. 打开"X 光报告"→ 看到窗口健康成绩单（驱逐事件、被淹没的关键块、冗余工具输出占用、修复建议）与 before/after 结论横幅。

## 7. 页面结构（<= 3 个主要视图，单页应用内切换）
- 视图 1 **Run X 光（主视图）**：顶部预算条 + 步骤时间线 + 右侧"上下文窗口 X 光"面板。
- 视图 2 **修复对比（Pin & Compare）**：基线 vs 修复的第 6 步窗口与结果并排/切换对比。
- 视图 3 **X 光报告（Report）**：窗口健康成绩单 + 结论横幅 + 修复建议清单。

## 8. 关键交互
- 顶部"基线 / 修复"场景切换（驱动整条轨迹与窗口数据变化）。
- 时间线步骤点击 → 联动右侧窗口 X 光。
- 窗口块 hover/点击 → 显示该块 token 数、来源、状态（保留 / 截断 / 驱逐）与原因。
- 预算条超阈值时可视化"溢出"，并高亮被牺牲的块。
- "驱逐 → 后果"因果连线：从被驱逐的策略块指向出问题的步骤。

## 9. 使用的 mock 数据
- 全部为手写 mock，位于 `src/data/scenario.ts`。
- 结构：一条 `AgentRun`，含 `budgetTokens`（窗口预算）与 `steps[]`；每步含 `action`、`risk`、`contextBlocks[]`（`{id,label,kind,tokens,status: kept|truncated|evicted, reason?}`）、可选 `evictionEvent`、`outcome`。
- 两个场景：`baseline`（策略被驱逐 → 第 6 步危险放款）与 `fixed`（Pin 策略 + 压缩工具输出 → 第 6 步安全暂停请求审批）。
- 不接任何真实模型 / 后端 / 外部 API。

## 10. 明确不做什么
- 不做登录 / 用户系统。
- 不接数据库 / 支付 / 外部私钥 / 真实 API / 真实 LLM 调用。
- 不做生产级后端；不做真实的 Agent 框架接入（真实产品的最大工程挑战，Demo 用 mock 规避并在文档中说明）。
- 不超过 3 个主要视图；全部数据 mock。

## 11. 成功体验标准
- 用户 3 分钟内能说出："它让我看见 Agent 每步上下文窗口里到底装了什么、什么被挤掉了、以及这如何导致了危险动作；用 Pin/压缩就能修复"。
- before/after 能让人**直观看到**"危险放款 → 被安全拦下"的增量价值。
- 首屏不空白，核心交互（选步联动、场景切换、报告）均可用，非空壳。
