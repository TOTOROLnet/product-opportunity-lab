## 1. 报告事实 vs 我的独立判断

| 维度 | 报告怎么说（参考资料） | 我作为创业者的独立判断 |
| --- | --- | --- |
| 审批粒度 | 高风险动作用"展示 payload → 人批准/拒绝"，工具设 Always allow / Needs approval / Blocked | 部分认同。但报告默认审批是**逐动作、逐工具**做的。我的判断：**风险不是单个工具决定的，而是从工具的"组合"里涌现的**。把"读客户库"设为 Always allow、把"发邮件"也设为 Always allow，各自都像低风险，**组合起来就是一条数据外泄链路**。逐工具审批看不见这一层。 |
| 权限模型 | Always allow / Needs approval / Blocked 是三态开关 | 我质疑三态开关的可组合性。它是**离散、无状态、按单工具**的策略；现实里危险来自"链路"（读敏感数据 → 外发；读表 → 无 WHERE 写库 → 不可逆）。缺一个把这些开关**放在同一张能力图上**、算出**涌现高危路径**的视图。 |
| 审批时机 | 审批发生在**动作触发的运行时**（Basedash/Banger Mail/Gavel 都在动作发生时拦） | 认同运行时审批必要，但它来得太晚且易疲劳：动作已到眼前才判断，且每条都要人看会**审批疲劳→橡皮图章**。真正缺的是**部署前 / 配置时**就能看清"我授予的这组权限能被组合成哪些高危链路"，从源头收敛权限面。 |
| Context Engineering / 控制面 | 报告把重点放在"证据层/记忆层/审批层" | 报告反复出现"权限边界 / MCP 工具组合 / Skills 编排"，却没有产品**把 Agent 的"能力面"（授予的工具集合）当作一等对象来做风险分析**。而"高风险工具调用的控制面"正是 `lab-focus.md` 明列重点——这里存在一个**被信号环绕、却无人正面做**的空位。 |

结论：报告的审批/权限叙事都停在**单动作、单工具、运行时**粒度。真正被忽视、且命中 lab 关注方向的**单点楔子**是——**把 Agent 授予的工具/MCP 权限集合建成一张"能力图"，自动发现"单看都安全、组合起来高危"的涌现链路（爆炸半径 / 可逆性 / 数据敏感度），并允许在部署前模拟策略变更、看高危链路如何收敛**。

## 2. 候选机会（至少 3 个，均为创新切入点，非照抄报告中的产品）

### 候选 A：Fusebox — Agent 能力"配电箱"：高危工具链路的部署前控制面
- 创新切入点：把 Agent 授予的工具/MCP 权限**集合**建成一张能力图（数据源 → 工具 → 外部副作用），自动识别**涌现的高危复合链路**（如"读客户 PII → 外发邮件 = 数据外泄"、"读表 → 无护栏写库 = 破坏性且不可逆"、"退款工具无金额上限 = 财务风险"），对每条链路给出**爆炸半径 / 可逆性 / 数据敏感度**评分；并提供**策略模拟器**——收紧某个工具权限或加一条"防火墙"规则后，实时看高危链路数如何下降（before / after）。
- 目标用户 / 场景：给生产 Agent 接一堆 MCP/工具的开发者、平台/安全工程师、对 Agent 行为负责的 AI PM——在**上线前**决定"该授予哪些权限、要不要拆链路"。
- 五维评分：痛点 5/5，机制新意 4/5，方向相关 5/5，一天可行性 5/5，启发价值 5/5 → 合计 **24/25**
  - （诚实校准：机制新意给 4——"能力图 / 攻击路径"范式在云 IAM 安全（如 BloodHound、PMapper、云攻击面工具）里已有先例；**把它迁移到 Agent/MCP 工具权限、并叠加"爆炸半径 + 可逆性 + 部署前策略模拟"是新的应用切入点**，而非全新算法。）

### 候选 B：Undo — Agent 高风险动作的"可逆性规划器"
- 创新切入点：不同于报告里"展示 payload → 批准"的审批范式，Undo 在动作执行**前**为每个写动作自动生成**回滚方案**并打**不可逆性分数**，把闸门从"你批不批"换成"**我们能不能撤销**"。对可逆动作放行、不可逆动作强制升级审批。
- 目标用户 / 场景：跑自动化写操作（发邮件、改库、退款、发消息）的 Agent 团队。
- 五维评分：痛点 4/5，机制新意 4/5，方向相关 5/5，一天可行性 4/5，启发价值 4/5 → 合计 **21/25**
  - （可逆性视角新颖，但真实落地需要为每类工具建回滚适配器，Demo 只能 mock 撤销方案，说服力略弱于 A。）

### 候选 C：Toolproof — MCP 工具契约漂移哨兵
- 创新切入点：MCP server 一旦升级工具 schema / 改变行为，依赖它的 Agent 会**静默失效**。Toolproof 给工具契约打快照，检测破坏性变更（字段删改、语义漂移、权限扩大），并映射出"哪些 Agent / Skill 会因此受影响"。
- 目标用户 / 场景：依赖多个第三方 MCP server 的 Agent 平台 / 团队。
- 五维评分：痛点 4/5，机制新意 3/5，方向相关 4/5，一天可行性 4/5，启发价值 3/5 → 合计 **18/25**
  - （契约/schema 漂移检测在 API 生态已常见，迁移到 MCP 有价值但机制新意有限，且受众偏窄。）

## 3. 最终选择

- 选中：**候选 A — Fusebox（Agent 能力配电箱 / 高危工具链路控制面）**，总分 **24/25**
- 为什么选它（相对 B、C 的优势）：
  1. **痛点最真实、最当下**：报告本身就在展示"越来越多 MCP/工具被接入 Agent、且用离散三态开关管权限"。任何给 Agent 接了三个以上有副作用工具的人，都直觉地担心"它会不会把 A 的数据通过 B 发出去"——但**没有工具帮他们把这种"组合风险"显性化**。
  2. **最适合轻量 Demo**：能力图 + 高危路径高亮 + 策略前后对比，天生是**可视化 + before/after** 题材，纯前端 + mock 权限拓扑即可把价值讲透（属 loop 的"抽象/基础设施类 → 模拟体验 + 价值可视化"型）。
  3. **与 lab 关注方向命中最准**：`lab-focus.md` 明列"MCP、Tool Use、**高风险工具调用的控制面**"，Fusebox 正中靶心；且与本仓库已有的 Gavel（动作层）、Contextlens（认知层）分属不同层，互补而非重叠。
  4. 相比 B（需回滚适配器、Demo 说服力弱）、C（机制新意与受众都弱），A 的独立性、当下性与可演示性都更强。

## 4. 深入论证（针对 Fusebox）

1. 目标用户：给生产 Agent 接入多个 MCP/工具（数据库、支付、邮件、Slack、内部文档、Computer Use 等）的**开发者 / 平台工程师 / 安全工程师 / 对 Agent 行为负责的 AI PM**。
2. 核心痛点（真实、高频、可付费）：Agent 的能力面在快速膨胀，权限却是**逐工具、离散、无状态**授予的。危险恰恰来自**组合**——数据外泄、破坏性写入、不可逆财务动作，往往是几个"单看无害"的工具被 Agent 串起来跑出来的。团队现在只能靠脑补，缺一张"我到底给了它多大爆炸半径"的图。
3. 现有产品为什么没解决好：Basedash / Banger Mail / Gavel 做的是**运行时、逐动作**审批（动作已发生才拦，且易疲劳）；Retrace 做**事后**重放（哪一步错）；scritty/Context.dev 做**记忆/上下文供给**；Contextlens（本仓库）做**上下文窗口可观测**。**没有一个把"授予的工具权限集合"当作一等对象，去发现组合出的高危链路，并允许部署前收敛权限面。**
4. 为什么适合做轻量 Demo：核心体验是"看能力图 → 看系统自动标红的高危复合链路（附爆炸半径/可逆性解释）→ 在策略模拟器里收紧一条权限/加一条防火墙 → 看高危链路数 before/after 收敛"，三视图、纯前端、全 mock 即可闭环，无需真实后端、真实凭证或模型。

## 5. 不照抄声明（四项必填）

1. **创新切入点是什么**：把 Agent 授予的**工具/MCP 权限集合**变成一张可分析的**能力图**，自动发现"单看安全、组合高危"的**涌现链路**，用**爆炸半径 / 可逆性 / 数据敏感度**量化，并支持**部署前的策略模拟（before/after）**从源头收敛权限面。
2. **解决的具体问题 / 场景**："我给这个 Agent 授的这一组工具，会不会被它组合成数据外泄 / 破坏性写入 / 不可逆财务动作的链路？"——这是接多工具 Agent 时人人担心、却无工具显性化的盲区。
3. **相对被参考产品的增量价值（差异点）**：
   - vs **Basedash Actions**：Basedash 是**逐动作展示 payload、逐工具设三态开关、运行时**审批；Fusebox 面向**权限集合的组合**，在**部署前**发现单工具设置永远暴露不出的**涌现高危路径**，并量化爆炸半径与可逆性。Basedash 决定"每个工具单独能不能用"，Fusebox 回答"这些决定组合成了什么"。
   - vs **Retrace**：Retrace 是**事后**重放已发生的失败轨迹（哪一步错）；Fusebox 是**事前**分析"这组权限能构成哪些危险路径"（还没跑就看清爆炸半径），关注**能力面**而非**执行轨迹**。
   - vs **PieterPost / Macuse（MCP 高危动作样本）**：它们是**提供**高风险 MCP 工具（现实邮寄 / 控制 macOS）；Fusebox 是**治理**当你把这类工具**和其他工具凑在一起**授予 Agent 时的组合风险。
   - vs **Gavel / Contextlens（本仓库旧示例）**：Gavel 做"动作发生时的人类审批"（动作层）；Contextlens 做"决策形成前的上下文可观测"（认知层）；Fusebox 做"**授权时的能力面组合风险**"（控制面/权限层）——三者是 Agent 生产化的三个不同层，互补而非同一层。
4. **为什么这不是照抄 X（本质区别）**：报告里没有任何产品把"Agent 的工具权限**集合**"当作一等对象、去做**组合链路的风险发现 + 爆炸半径/可逆性量化 + 部署前策略模拟**。报告中的产品要么在单动作运行时审批，要么在事后重放，要么在提供工具。Fusebox 的核心创新（能力图 + 涌现高危链路 + 爆炸半径/可逆性评分 + 策略 before/after）在报告任一产品中都不存在，因此不是克隆。

## 6. 不确定性与风险

- 需求侧风险：很多团队的 Agent 目前工具还很少（1–2 个），组合风险感知不强，付费意愿可能滞后于痛点。对冲：随着 MCP 生态扩张与"Skills 编排多步流程"普及（报告已见此趋势），工具数会快速增长，组合风险会从"隐忧"变"事故"。
- 落地 / 技术风险：真实产品要拿到"某 Agent 实际被授予的完整工具/MCP 清单 + 每个工具的读/写/外发语义 + 数据敏感度标注"，这依赖对接各 Agent 框架与 MCP 注册表，标准化难；"爆炸半径/可逆性"评分若太主观会失去信任。Demo 用 mock 拓扑与显式评分规则规避，但这是真实落地最大的工程/信任挑战。
- 我可能判断错的地方：如果多数高危动作最终都会被"运行时逐动作审批"兜住，那"部署前组合分析"可能被视为"锦上添花"而非"必需"。需要用真实的组合型事故案例证明"运行时审批来得太晚 / 会被审批疲劳绕过"。
