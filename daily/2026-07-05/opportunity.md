## 1. 报告事实 vs 我的独立判断

| 维度 | 报告怎么说（参考资料） | 我作为创业者的独立判断 |
| --- | --- | --- |
| Adam 的核心创新在哪 | 在专业软件内用自然语言**改真实对象**、并保持产物可编辑、提供可审查 diff | 部分认同，但我认为 Adam 真正稀缺、又最难做对的不是"能改"，而是"**改完之后，人怎么在几十秒内看清它到底改了什么意义、有没有破坏设计意图**"。"可审查 diff"被 Adam 当成一句 feature 承诺，但它其实是一个**独立、跨领域、比 CAD 大得多的空位**。 |
| "diff / 审批"该由谁做 | 报告建议"拆解 Adam 的 **CAD** diff / 审批机制" | 我质疑把它锁死在 CAD。**"Agent 编辑结构化产物"是个通用范式**：CAD 的 feature tree、云基建的 Terraform / K8s、数据库 schema / migration、CI 配置、状态机 / 工作流定义——都是"结构化产物"。它们共享**同一个痛点**：Agent 改完，**行级 diff（git 那种）看不出语义**，更看不出"是否破坏了意图 / 不变量"。真正的机会是**跨领域的"结构化产物语义评审层"**，CAD 只是最生动的一个实例。 |
| Agent 编辑的核心风险 | "破坏设计意图" | 强烈认同，且这正是被行级 diff **系统性隐藏**的风险。一个 Agent 可以做出"**行级看起来很小、语义上很危险**"的改动：删掉一个 `NOT NULL` 约束、把安全组 CIDR 从 `10.0.0.0/8` 悄悄放宽到 `0.0.0.0/0`、删掉一个被下游 feature 依赖的圆角、改掉一个改变默认行为的默认值。**+3/-1 行的 diff 骗过 reviewer，语义上却是一次高危改动**。 |
| 可控性该做成什么 | Termi 做 run trace / 暂停 / 审批 / 文件锁（**过程**可视化） | 认同"可控性要前台化"，但我认为 Termi / 报告的落点全在**过程**（它读了什么、跑了什么命令、花了多少钱）。而**评审真正的决策依据是"产物的 delta 有没有问题"**，不是"过程好不好看"。过程可视化解决"它在忙什么"，**没有解决"它交出来的这版结构化产物，语义上安不安全"**。这是一个被信号环绕、却无人正面做的下游空位。 |
| 现有 diff / 评审工具 | 报告未展开 | 现状是：行级 diff（GitHub / IDE）语义盲；少数**单领域**语义工具存在（`terraform plan`、schema diff 工具、CAD 自带比较）。但**没有一个以"评审 Agent 改动"为中心、跨产物类型、把'语义 delta + 意图/不变量风险'统一呈现**的评审层。Agent 时代把这个空位从"nice to have"逼成了"必须有"——因为改动量暴涨、且改动者是不会主动坦白副作用的模型。 |

结论：报告的叙事集中在"Agent **进入**专业现场 + **过程**可控（trace / 审批 / 成本）"。真正被忽视、又高频命中痛点的**单点楔子**是——**当 Agent 编辑一个结构化产物时，把改动从"行级 diff"升级为"语义 diff（改了哪些实体 / 关系 / 属性），并对每个改动跑意图 / 不变量核验，把'行级看着很小、语义上很危险'的改动顶到最显眼处"，让 reviewer 在几十秒内看清"它真正改了什么意义、哪几处可能破坏了设计意图"。**

## 2. 候选机会（至少 3 个，均为创新切入点，非照抄报告中的产品）

### 候选 A：Contour — 结构化产物的"语义 diff + 意图/不变量评审"层（看清 Agent 改动的真实轮廓）
- 创新切入点：不做审批、不做过程可视化、不核验自述文字。取 Agent 对**任意结构化产物**（CAD feature tree / 云基建配置 / 数据库 schema / CI 配置 / 工作流定义）的 **before / after**，把它解析成**语义模型（实体 / 关系 / 属性 / 约束）**，计算一份**语义 diff**（新增 / 删除 / 修改的是**实体与关系**，而不是行），再对每个语义改动跑一组**意图 / 不变量核验**（如"权限是否被放宽 / 约束是否被移除 / 引用是否被打断 / 默认值是否改变了行为"），把 **Critical / Warn / Info** 风险按严重度排序，并特别把"**行级 diff 里看不见、语义上却高危**"的改动打上标记顶到最前。给出 before/after 评审判定：**Agent 说"改动很小 ✅（+3/-1 行）" → Contour 说"2 处 Critical 意图风险 / 1 处约束被破坏，先看这里"**。
- 目标用户 / 场景：用 Cursor / Copilot / Devin / Claude Code 等让 Agent 改**结构化配置/schema/基建/CAD** 的开发者、reviewer、平台 / SRE / DBA、以及对 Agent 交付负责的 AI PM——在 **accept / merge 之前**几十秒定位"哪几处改动语义上危险"。
- 五维评分：痛点 5/5，机制新意 4/5，方向相关 5/5，一天可行性 5/5，启发价值 5/5 → 合计 **24/25**
  - （诚实校准：机制新意给 4——单领域语义 diff（terraform plan、schema diff、CAD 比较）已有先例；**把它抽象成跨产物类型、以"评审 Agent 改动 / 狙击破坏设计意图"为中心的统一评审层，是新的应用切入点与产品框架**，而非全新算法。）

### 候选 B：Relay — 并行多 Agent 的"写入冲突仲裁 / 争用可视化"
- 创新切入点：受 Termi"文件锁 / 多 Agent 冲突控制"启发，但聚焦一个更窄的痛：多个 coding agent 并行改同一仓库时的**写入争用**——谁在改哪些文件、哪些改动互相覆盖、如何按依赖顺序仲裁与合并。
- 目标用户 / 场景：跑并行 Agent 编队的团队 / 平台。
- 五维评分：痛点 4/5，机制新意 3/5，方向相关 5/5，一天可行性 4/5，启发价值 3/5 → 合计 **19/25**
  - （文件锁 / 合并冲突解决是很成熟的老问题；且"并行多 Agent"当下受众仍偏窄，多数人一次只跑一个 Agent。）

### 候选 C：Meter — Agent run 的"成本护栏"（预算预估 + 失控熔断 + 逐步花费归因）
- 创新切入点：受 Termi"实时成本"启发，做 run 前预算预估、逐 step token/$ 归因、以及"失控自动熔断"的护栏，而不只是事后一张成本仪表盘。
- 目标用户 / 场景：为 Agent 平台账单负责的团队 / FinOps。
- 五维评分：痛点 4/5，机制新意 3/5，方向相关 4/5，一天可行性 5/5，启发价值 3/5 → 合计 **19/25**
  - （FinOps 仪表盘已很多；逐 step 归因 + 预算护栏略新，但"成本"当下更多是烦恼而非高危阻断，付费意愿弱于 A 的"防止把危险改动 merge 进生产"。）

## 3. 最终选择

- 选中：**候选 A — Contour（结构化产物的语义 diff + 意图/不变量评审层）**，总分 **24/25**
- 为什么选它（相对 B、C 的优势）：
  1. **痛点最真实、最当下、最高频，且被报告当日最高分产品（Adam 17/18）与其点名的失败模式直接背书**："Agent 改结构化产物 → reviewer 只有一份语义盲的行级 diff → 危险改动被 +3/-1 行伪装成'小改动'混过评审"是每天都在发生的事。报告亲口说 Adam 的最大风险是"**破坏设计意图**"，而破坏设计意图恰恰是**行级 diff 系统性看不见**的那一类。随 Agent 交付量上升，这个矛盾只会越来越痛。
  2. **最适合轻量 Demo**：核心体验 = "选一个 Agent 改动 → 看行级 diff（骗人的'小改动'）→ 切到语义 diff（改了哪些实体/关系）→ 意图/不变量风险按严重度排序 → before/after 评审判定"，天生是**语义可视化 + before/after 对比 + 可点击核查**题材，纯前端 + mock（before/after 产物 + 确定性语义解析规则 + 预置不变量）即可把价值讲透（属 loop 的"抽象/基础设施类 → 模拟体验 + 价值可视化"型）。
  3. **命中 lab 关注方向最准、且与本仓库旧产物分属不同层**：命中"Coding Agent / AI 原生生产力 / Developer Tools / 高风险工具调用的控制面"；且它是**产物 delta 的"语义评审层"**——与 Gavel（动作发生时审批）、Contextlens（决策前上下文）、Retrace（过程重放）、Fusebox（授权时权限组合）、Attestor（自述**文字**↔证据核验）、以及 Termi（**过程**可视化）都不同层，互补而非重叠。
  4. 相比 B（老问题 + 受众窄）、C（付费意愿弱 + 机制平），A 的独立性、当下性、付费意愿与可演示性都更强。

## 4. 深入论证（针对 Contour）

1. 目标用户：**要为"Agent 改的结构化产物"签字的人**——用 Agent 改 Terraform / K8s / DB schema / CI 配置 / CAD 的开发者、code reviewer、SRE / 平台 / DBA、以及对 Agent 交付质量负责的 AI PM / 团队 lead。他们的共同动作是"在 merge / apply / accept 之前，判断这次结构化改动安不安全"。
2. 核心痛点（真实、高频、可付费）：Agent 给出的改动，**reviewer 手里只有一份行级 diff**。行级 diff 天生**语义盲**：它告诉你"这几行变了"，不告诉你"一个约束被删了 / 一个网络被放开了 / 一个下游依赖被打断了 / 一个默认值改变了运行时行为"。于是 reviewer 只有两个坏选项：**要么信任那份看起来很小的 diff（可能放进一次高危改动），要么逐字段人肉重审整个产物（抵消 Agent 的效率）**。"评审结构化改动的语义安全性"正处在团队愿意付费的价值线上（code review、IaC 策略扫描、schema 迁移审查工具都在这条线上）。
3. 现有产品为什么没解决好：**Adam** 把"可审查 diff"当成 **CAD 内部**的一句 feature，不跨领域、也不以"狙击破坏设计意图"为中心；**Termi** 做**过程**可视化（读写/命令/成本/检查点），不碰"产物 delta 的语义安不安全"；**Gavel/Basedash** 在动作**发生时**审批（运行时闸门），不看结构化 delta；**Attestor**（本仓库旧产物）核验 Agent **自述文字**是否有证据支撑，**不解析产物本身的语义**；行级 diff（GitHub/IDE）语义盲；单领域工具（terraform plan / schema diff）各自为战、且不是为"评审 Agent 改动"设计。**没有一个把"任意结构化产物的语义 delta + 意图/不变量风险"做成统一的评审层。**
4. 为什么适合做轻量 Demo：核心体验是"**看骗人的行级 diff → 切到语义 diff → 意图/不变量风险高亮 → before/after 评审判定**"，两三个跨领域场景（云基建 / DB schema / CAD feature tree）、纯前端、全 mock（预置 before/after 产物 + 确定性语义解析 + 显式不变量规则）即可闭环，无需真实后端、真实凭证或模型。

## 5. 不照抄声明（四项必填）

1. **创新切入点是什么**：把 Agent 对**结构化产物**的编辑，从"行级 diff"升级为"**语义 diff（实体/关系/属性/约束的增删改）+ 意图/不变量核验**"，并专门把"**行级看着很小、语义上却高危**"的改动（破坏设计意图 / 放宽权限 / 移除约束 / 打断引用 / 改变默认行为）顶到最显眼处，用 **before/after 评审判定**量化"Agent 说的'小改动'"与"真实语义风险"的差距。**跨产物类型（CAD / IaC / schema / 配置）通用**，不锁死在任何单一领域。
2. **解决的具体问题 / 场景**："Agent 改了这份配置/schema/模型，diff 看起来就 +3/-1 行，我能放心 merge 吗？"——Contour 直接指出"这处约束被删了 / 这处网络被放开了 / 这处下游依赖被打断了，行级 diff 里看不见，先看这三处"，把评审从"看行数信任"变成"看语义信任"。
3. **相对被参考产品的增量价值（差异点）**：
   - vs **Adam CAD Copilot**：Adam 是**在 CAD 内部替你改**对象的**垂直 authoring copilot**（生产改动），"可审查 diff"只是它的一句 feature、且**锁死在 CAD**。Contour **不改任何东西、不绑定任何领域**，是一个**跨产物类型的评审镜片**：消费**任意** Agent 交出的结构化产物 before/after，专门做"语义 delta + 意图/不变量风险"呈现。Adam 答"帮我在 CAD 里改这个零件"，Contour 答"不管是谁改的、改的是什么类型的产物，这次改动语义上安不安全、有没有破坏设计意图"。二者是**authoring vs review**、**vertical vs cross-domain** 的不同层，长期互补。
   - vs **Termi Protocol**：Termi 可视化**执行过程**（读写/命令/成本/检查点/文件锁，且套了 3D 皮肤）；Contour **完全不看过程**，只看**产物 delta 的语义**。Termi 答"它在忙什么、花了多少钱"，Contour 答"它交出来这版产物，语义上改了什么、危不危险"。对象不同（过程 vs 产物）。
   - vs **Attestor（本仓库旧产物）**：Attestor 把 Agent 的**自然语言自述**拆成 claim，去和证据 artifact 连线，核验"它说的有没有证据"；它**不解析产物本身的语义结构**。Contour **无视 Agent 的任何叙述**，直接解析**结构化产物的语义 delta** 并跑不变量核验——即使 Agent 老实汇报，它也常常不会主动坦白"我顺手放宽了这个约束"这类副作用，而这恰恰是 Contour 的主场。Attestor 答"它有没有说谎"，Contour 答"不管它说没说，这次改动本身语义上破没破坏东西"。
   - vs **Fusebox / Gavel / Basedash（授权 & 运行时审批）**：它们在动作**发生前/时**决定 approve/deny（权限组合 / 运行时闸门）；Contour 在 Agent **改完产物之后**做**语义级 delta 评审**，不是权限闸门。
   - vs **行级 diff（GitHub / IDE）与单领域工具（terraform plan / schema diff）**：行级 diff 语义盲；单领域工具各自为战、且不是为"评审 Agent 改动 / 狙击破坏设计意图"设计。Contour 的创新是**跨产物类型的统一语义评审框架 + 意图风险狙击 + before/after 评审判定**。
4. **为什么这不是照抄 X（本质区别）**：报告里**没有任何产品**把"任意结构化产物的**语义 delta + 意图/不变量风险**"当作一等对象做**跨领域评审层**。报告中的产品要么在 CAD 内部**生产**改动（Adam），要么在可视化**执行过程**（Termi），要么在拼**个人 runtime**（Flowly），要么在做**采购/AP 流程**（Saldor）。Contour 的核心创新（跨产物类型的语义 diff + 意图/不变量核验 + "行级看不见的高危改动"狙击 + before/after 评审判定）在报告任一产品中都不存在，因此不是克隆；它正是报告为 Adam 点名的失败模式（"破坏设计意图"）与作者所说"可审查 diff / 需要人接管"的一个**可落地、且比 CAD 大得多的通用原语**。

## 6. 不确定性与风险

- 需求侧风险：有人会说"配置/schema 的风险，policy-as-code（OPA / Conftest）/ CI 检查不是已经在扫了吗？"对冲：那些工具是**per-domain、规则预写死、且面向 CI 通过/失败**，不面向"**人类 reviewer 在 accept 前理解这次改动的语义**"；且它们**不呈现语义 diff**（改了哪些实体/关系），只给通过/失败。Contour 的差异化正是"**跨领域 + 语义可视化 + 面向人类评审决策 + 专治 Agent 交付的隐性副作用**"。真实产品应与 policy 引擎互补（把它们的判定也纳进 findings），而非替代。
- 落地 / 技术风险：真实产品要为**每种产物类型**写可靠的解析器与不变量库（CAD、Terraform、K8s、SQL schema 各不相同），这是重活；不变量判太松会漏报、判太严会满屏告警造成新疲劳；判定依据若不透明会失去信任。Demo 用**确定性解析 + 显式不变量规则 + 每条 finding 附"为何行级 diff 看不见 + 判定依据"**规避，但这是真实落地最大的覆盖面/准确率挑战。
- 我可能判断错的地方：也许多数团队更想要"**直接帮我把危险改动挡在 CI 外（gate）**"而不是"**给人看语义评审再由人决定**"。若如此，Contour 需要从"评审镜片"进一步长出"可配置 gate / 策略"能力（与 policy-as-code 融合）。这说明 Contour 与 policy 引擎长期是**互补**关系——更印证它不是照抄某个单一产品。

- 最高分：**24/25**
- 是否达到 16/25 门槛：**是**
